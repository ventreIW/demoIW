# Retrospective: S4.4 — Score Explanation

**Dates:** 2026-07-21 → 2026-07-21 (design and plan on 2026-07-21, implementation same day)
**Estimated:** S (3 tasks) | **Actual:** S — 3 tasks, one defect found mid-story and fixed in place

## Summary

Every collectability score now carries a one-sentence Spanish explanation naming what drove it,
built from the model's own exact contributions. 43 new tests; backend 217 → 260.

The story delivered what RF-02.3 asked for. What it actually taught was narrower and more useful:
**a set of rules can be individually correct and jointly produce nonsense**, and the only way to
find that is to read the output.

## What went well

- **Measuring before designing paid off again.** The 0.885 correlation between the two ageing
  contributions, and the 26.2% of clients holding both in their top tier, turned "we should
  probably group these" into a specification with a number behind it.
- **Grouping before ranking was worth isolating as its own test.**
  `test_grouping_happens_before_ranking` constructs the exact case where late grouping fails —
  `pct_invoices_settled` at 0.40 beats either ageing feature individually, but summed ageing
  reaches 0.49 and should lead. Without that test, a plausible implementation would have quietly
  named the wrong lead factor.
- **The legacy sweep was executed, not intended.** `_placeholder_explanation` is gone and a test
  asserts both that the attribute is absent and that the name does not appear in the module.
- **Determinism was designed in.** Ties break by factor key rather than dict order, verified over
  five runs. Without it, clients with two equally-weighted drivers would have produced
  intermittently different explanations — a determinism failure that passes most of the time.

## What to improve

- **I repeated a lesson I had written down one story earlier.** s4.3's retrospective says *"tests
  that read documentation instead of behaviour fail for the wrong reasons"*. The AC #12 boundary
  test was then written as a substring grep for `"sklearn"` and immediately failed on a docstring
  *describing the rule* and a module importing our own `SklearnScorer` class. Recording a lesson
  is not the same as applying it — the same gap as the pull-board rule I wrote and then broke.
- **The design specified phrasing as "render the business fact" and stopped there.** It never
  asked what a fact *means* when the contribution that selected it is positive. That omission
  produced the story's central defect and was only visible on real output.
- **`_phrase` is a long if-chain.** It is readable and every branch is tested, but it will not
  scale gracefully if E5 wants per-channel phrasing. Left as-is deliberately — refactoring for a
  requirement that does not exist yet is the YAGNI violation the design gates exist to prevent.

## Heutagogical Checkpoint

1. **What did I learn?**
   That correctness at the rule level does not compose into correctness at the output level. Every
   rule in T2 was right — group ageing, rank by magnitude, filter by direction, render the fact —
   and together they produced *"Buen perfil de pago: facturas con hasta 40 días de atraso"*. The
   missing idea was that a fact's *meaning* depends on the sign of the contribution that surfaced
   it: 40 days is favourable for a client whose peers are at 200, and damning for one whose peers
   are at 10.

2. **What would I change about the process?**
   Read generated output during **design**, not at the end of implementation. I prototyped the
   *numbers* in s4.3 and s4.4 and caught real problems both times, but I did not prototype the
   *text*. Ten lines rendering three example sentences from measured contributions would have
   exposed the framing gap before any code existed.

3. **Framework improvements?**
   - `rai-story-design`'s examples section drives code generation, and mine contained the defect:
     the example output showed `"facturas con hasta 78 días de atraso"` for a low score only, so
     the positive case was never imagined. The skill should prompt for **an example per branch of
     the output space**, not per happy path.
   - Two stories running, two structural ACs written as prose and violated or mis-tested. This is
     now parked twice; it deserves a real mechanism rather than a third note.

4. **More capable of now?**
   Writing explanation layers over linear models — specifically, recognising that attribution
   requires three separate correctness conditions: stable coefficients (s4.3's `C=0.01`), grouped
   correlated features (T1), and sign-aware phrasing (T2). Any one missing produces confident,
   wrong prose.

## Improvements applied

- Phrasing made direction-aware across all six factor templates, with three regression tests each
  naming the sentence that motivated it.
- Boundary test rewritten from substring grep to `ast` import parsing, then **verified
  non-vacuous** by lifting its exclusion and confirming it reports the two legitimate importers.
- Pull board updated in the same commit as the work, correcting yesterday's failure to update it
  at transitions.

## Patterns added

| Pattern | Type |
|---|---|
| Templated user-facing text must frame a fact by the sign of its contribution, not just render its value | technical |
| Print and read generated user-facing output before declaring a story done | process |
| Verify a boundary or guard test is non-vacuous before trusting it | process |
| Group correlated features before ranking, never at render time | technical |

## Carried forward

- **The Spanish register is unvalidated by anyone who does collections.** Every automated check
  passes, but whether an officer finds this natural is a human judgement nobody has made yet.
  Worth five minutes with Gustavo before the demo.
- **The generator's exponential tail produces extreme ageing** (679, 824 days). Arithmetically
  correct, possibly implausible for a demo portfolio — a generator-tuning question, not an
  explanation bug.
- E5's case detail view consumes `Score.explanation` directly; no further API is needed from here.
