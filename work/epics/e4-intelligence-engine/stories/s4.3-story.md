# Story: Collectability Scoring Model

**Owner:** Rodrigo
**Branch:** `story/s4.3/collectability-model`

## User Story

As a **collections manager**,
I want every client in a scenario scored 0–100 on how likely their outstanding balance is to be
collected, with a High/Medium/Low band,
so that the prioritization engine can rank a portfolio by expected recoverable value instead of
by gut feel.

## Acceptance Criteria

### Scenario: A scenario can be scored end to end

```gherkin
Given an active scenario with at least 20 clients holding an outstanding balance
When the scoring engine runs
Then every one of those clients receives a score between 0 and 100
And each score carries a High, Medium or Low category
```

### Scenario: The model beats a naive baseline

```gherkin
Given a training set built from a 400-client scenario
When the model is trained and evaluated on held-out clients
Then ROC-AUC is materially above 0.5
And balanced accuracy exceeds the majority-class baseline
```

### Scenario: Raw accuracy is not used as the success metric

```gherkin
Given a training set whose positive class is roughly 80%
When the model is evaluated
Then the reported metrics include ROC-AUC and balanced accuracy
And a model predicting the majority class for every client fails the quality gate
```

### Scenario: Coefficient direction sanity check

```gherkin
Given a trained model
When its coefficients are inspected
Then days_overdue_max carries a negative coefficient
And pct_invoices_settled carries a positive coefficient
```

### Scenario: Scoring is reproducible

```gherkin
Given a scenario generated with seed 42
When it is scored twice
Then both runs produce identical scores for every client
```

### Scenario: A scenario too small to score fails clearly

```gherkin
Given a scenario with 10 clients
When scoring is requested
Then the caller receives an explicit "scenario too small to score" error
And the error states how many scoreable clients were found and how many are needed
```

### Scenario: Small scenarios cannot be generated in the first place

```gherkin
Given a request to generate a scenario with 10 clients
When GenerationParams is validated
Then it is rejected because client_count is below the scoreable minimum
```

### Scenario: The leakage guard still holds at the model boundary

```gherkin
Given a trained model
When its feature names are inspected
Then payment_history_pattern is absent
And no sector column is present
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|---|---|---|
| Client: 78 days overdue max, 2 of 7 invoices settled, partial payments | score | Low score (< 40), category `low` |
| Client: 0 days overdue, 6 of 6 settled, no partials | score | High score (> 70), category `high` |
| 400-client retail scenario, seed 42 | train + evaluate | ROC-AUC > 0.65, balanced accuracy > 0.5 |
| A model predicting all-positive | evaluate | Fails the quality gate despite ~0.80 raw accuracy |
| Trained model | inspect `coef_` | `days_overdue_max` < 0, `pct_invoices_settled` > 0 |
| Same scenario, seed 42, scored twice | score | Identical scores |
| 10-client scenario | score | `InsufficientTrainingDataError` naming counts |
| `GenerationParams(client_count=10)` | validate | `ValidationError` — below scoreable minimum |

## Notes

- Implements **ADR-007**: `scikit-learn` `LogisticRegression`, trained **on demand per scenario**,
  no persisted artifact. Access sits behind `IScoringPort` so the model can be replaced without
  touching use cases — the same guardrail that keeps the backend provider-agnostic for LLMs.
- **Metric choice is forced by measurement, not preference.** s4.2 measured the positive class at
  0.75–0.86 by sector, so majority-class already scores that as raw accuracy. ROC-AUC and
  balanced accuracy are the gate (ADR-006 D3).
- **Decisions taken 2026-07-21 before design:**
  - `sector` is dropped from the design matrix — it is constant within a scenario, so its
    one-hot column has zero variance and carries no signal.
  - Scenarios too small to score are prevented at generation (`client_count` minimum raised)
    **and** rejected with a clear error at scoring. Defence in depth.
- **Score persistence is s4.9 (Nano)** and is not yet merged. This story builds training,
  scoring, evaluation and categorization behind `IScoringPort`; persisting `Score` entities is
  wired once `IScoreRepository` exists.
- Category thresholds must be justified against the observed score distribution, not chosen by
  feel — an explicit epic acceptance-gate item.
- A free correctness canary from the epic design: if `days_overdue_max` comes out with a
  *positive* coefficient, the bug is upstream in the labeller, not in the model.
