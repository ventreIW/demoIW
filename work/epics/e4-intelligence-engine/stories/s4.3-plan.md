## Execution order

```
T1 (drop sector) ──→ T2 (port + scorer) ──┬──→ T3 (categorize + evaluate) ──┐
                                          │                                 ├──→ T4 ──→ T5
                                          └─────────────────────────────────┘
```

**Rationale.** Risk-first *within* the dependency graph. T1 is trivial but must precede
everything, since the design matrix shape is an input to the scorer. T2 carries the real risk
(Finding 1's silent sign inversion) and comes as early as its dependency allows. T3 depends on
T2 only for realistic score inputs. T5 is the gate.

**Dependency graph is acyclic.** T1 → T2 → {T3} → T4 → T5.

## Risks

| Risk | Mitigation |
|---|---|
| **AUC gate fails at T5** — s4.4/s4.5/s4.6 all rest on it | Stop and revise ADR-007. The prototype measured 0.605–0.819 at n=400, so a failure means something changed, not that the gate is wrong |
| Scaling regressed silently, inverting coefficients | Two independent tests: the scaler is present, and the coefficient signs are correct |
| Someone reports balanced accuracy at 0.5 and concludes the model is broken | D1's consequence is written into the design, and T3 measures balanced accuracy at the category cutoff instead |
| `sector` drop breaks s4.2's tests | T1 updates them deliberately in the same commit |
| Retail's weaker AUC surfaces in the demo | Recorded in T5; choose the demo sector deliberately |
| Persistence gap while s4.9 is unmerged | Use case returns entities; only the repository call is deferred |
