# s4.3 Design — Collectability Scoring Model

**Owner:** Rodrigo · **Complexity:** Complex (6 components, 1 new dependency, HIGH RISK — M1 go/no-go)
**Branch:** `story/s4.3/collectability-model`

## Problem

s4.2 produced a labelled training set. Nothing yet turns it into a score, and E4's entire value —
prioritization, explanation, the executive view — hangs off a number that must be both *ranked
correctly* and *interpretable*.

## Value

Every client with an outstanding balance gets a 0–100 score meaning "probability this balance is
collected within 90 days", plus a High/Medium/Low band. That is the input s4.5 ranks by and s4.4
explains.

## Prototype results (measured before designing)

Applying s4.2's retrospective lesson — *run the code, don't only read it*. A throwaway model was
trained against real generated scenarios during design. Three findings changed this design.

### Finding 1 — features must be scaled

Unscaled, `lbfgs` failed to converge at `max_iter=1000` (`outstanding_amount` ≈ 10⁴ against
`pct_invoices_settled` ≈ 10⁰) and produced **wrong coefficient signs** — `pct_invoices_settled`
came out negative. With `StandardScaler` the signs are correct:

| Feature | Coefficient (retail, n=1500, scaled) |
|---|---:|
| `pct_invoices_settled` | **+0.757** |
| `has_partial_payments` | +0.353 |
| `days_overdue_max` | **−0.295** |
| `outstanding_amount` | +0.260 |
| `avg_days_late_historical` | −0.197 |

The epic design's canary passes: `days_overdue_max` negative, `pct_invoices_settled` positive.
The story scope said "add scaling only if convergence requires it" — it does.

### Finding 2 — ROC-AUC across seeds, sectors and sizes

Five seeds per cell, no `class_weight`, scaled:

| Sector | n=100 | n=200 | n=400 | n=1000 |
|---|---|---|---|---|
| manufacturing | 0.393–1.000 | 0.612–0.872 | 0.681–0.819 | 0.693–0.792 |
| retail | **0.286–0.750** | 0.559–0.819 | 0.605–0.710 | 0.666–0.707 |
| professional_services | 0.364–1.000 | 0.637–0.760 | 0.692–0.806 | 0.694–0.836 |

**At n=100 the estimate is noise** — worse than chance to perfect, because ~100 clients yields
~60 usable and an ~18-row test set. From n≥200 it stabilises: minimum observed 0.559, mean 0.698.

Retail is consistently weakest — its pattern mix has the highest `partial` weight (0.20), and
partial payers sit mid-gradient, so the classes overlap more.

### Finding 3 — score distribution (retail, n=1000)

| p0 | p25 | p50 | p75 | p100 |
|---:|---:|---:|---:|---:|
| 4.1 | 40.3 | 53.4 | 67.5 | 90.1 |

Well spread, no clumping — the fixed 40/70 cutoffs land near natural quartiles.

## Decisions

### D1 — Calibrated probabilities, not `class_weight="balanced"`

No `class_weight`, so `predict_proba` remains a genuine estimate and **score = P(collected within
90 days) × 100**. A director reading 34 gets "34% likely to pay", which E6's executive panel needs.

`class_weight="balanced"` measured better on balanced accuracy (0.52–0.88 against 0.48–0.69) but
destroys that meaning: the probabilities stop estimating anything real.

**Consequence, stated so nobody reports it naively:** at a 0.5 cutoff on an 80/20 problem the
model calls almost everyone positive, so balanced accuracy sits near 0.5. That is a property of
the threshold, not of the model. **Gate on ROC-AUC**, which is threshold-free and measures
ranking — exactly what s4.5's prioritization consumes.

### D2 — Fixed absolute category thresholds

| Category | Score | Observed share |
|---|---|---:|
| Low | < 40 | ~25% |
| Medium | 40–70 | ~50% |
| High | > 70 | ~25% |

Fixed rather than per-scenario terciles so the same client gets the same band in any portfolio,
and so s4.6's rescore-after-contact moves a client's band only when that client changes. Terciles
would make every client's band depend on everyone else's.

### D3 — The quality gate

- **ROC-AUC ≥ 0.60** on held-out clients — clears the observed floor at n≥200 (0.559 at the very
  worst) while sitting well above chance.
- **Evaluated only when the test set holds ≥ 30 clients.** Below that, report
  `evaluation: insufficient_data` rather than a number. An 18-row AUC of 0.286 or 1.000 is noise,
  and shipping either as a quality signal would be worse than admitting the gap.
- **Balanced accuracy is reported at the chosen category cutoff, not at 0.5**, alongside the
  majority-class baseline for context.

### D4 — Minimum scenario size lives in the frontend form

Per decision: raise the form's `min` from 10 to 50 and return an explicit error when scoring a
too-small scenario. The API and `GenerationParams` stay permissive, so E3's fast integration
tests (3–5 clients) are untouched.

> ⚠️ **Measurement suggests 50 is too low.** Finding 2 shows n=100 still produces AUC between
> 0.286 and 1.000. For a credible demo the form minimum should be **200**, not 50. Flagged for
> revision — the decision was taken before this data existed.

### D5 — `sector` dropped from the design matrix

Per decision, and confirmed by the prototype: constant within a scenario, zero variance.
`CATEGORICAL_COLUMNS` is emptied but kept as a named constant for eventual cross-scenario work.

### Legacy sweep

**Nothing becomes orphaned — net-new.** No prior scoring implementation exists. `Score`,
`ScoreCategory` and the mappers are E2 assets being *used* for the first time, not superseded.

### Drift risk

`governance/drift-hotspots.json` absent — ranking skipped. **AG2 (clone amplification):** the
`sector_` column drop must happen in exactly one place; putting it in both `BuildTrainingSet` and
the scorer would let them diverge. Handled by removing `sector` from the design matrix at source
(s4.2's `_design_matrix`), so the scorer needs no filtering at all.

## Approach

| Component | Change | Purpose |
|---|---|---|
| `requirements.txt` | modify | `scikit-learn` (ADR-007) |
| `app/ports/scoring_port.py` | create | `IScoringPort` — `fit`, `predict_scores`, `feature_contributions` |
| `app/adapters/scoring/sklearn_scorer.py` | create | `StandardScaler` + `LogisticRegression` pipeline |
| `app/domain/value_objects/scoring_result.py` | create | Scores, categories, evaluation metrics |
| `app/application/services/score_categorizer.py` | create | Score → `ScoreCategory` at the D2 cutoffs |
| `app/application/use_cases/score_scenario.py` | create | Build training set → fit → score → `Score` entities |
| `app/domain/value_objects/client_features.py` | modify | Empty `CATEGORICAL_COLUMNS` (D5) |
| `app/application/use_cases/build_training_set.py` | modify | Drop the one-hot block (D5) |

**Lean gates.** KISS: one sklearn `Pipeline`, no tuning, no grid search. DRY: categorization is
one function, used by scoring and by future rescoring. YAGNI: no model persistence, no
hyperparameter search, no cross-validation beyond the single held-out split — add only if the
gate fails. MVP: this story ends with `Score` entities in memory.

**Supply chain — `scikit-learn`:** mature (2010–, 1.9.0), one of the most-downloaded Python
packages, active multi-maintainer development, no yank history of note. Pulls `scipy` (~60 MB),
accepted in ADR-007. **Isolation:** imported only in `adapters/scoring/`; nothing else in the
codebase may import it, which is what `IScoringPort` is for.

**Missing-data strategy:** inherited from s4.2 — clients with no outstanding balance never enter
the training set. Scoring covers exactly the clients that were labellable; the rest have no score,
reported as a count rather than defaulted to 0 (a 0 would read as "certain not to collect").

## Acceptance criteria

Refines the Gherkin in `s4.3-story.md`.

**MUST**

1. Every client with an outstanding balance receives a 0–100 score and a `ScoreCategory`
2. **ROC-AUC ≥ 0.60** on held-out clients when the test set has ≥ 30 rows
3. Test sets below 30 rows report `status='insufficient_data'` with `roc_auc=None` — never a number
4. `days_overdue_max` coefficient negative; `pct_invoices_settled` positive (the pipeline canary)
5. Features are scaled — an unscaled pipeline is a defect, not a style choice
6. Score is `predict_proba × 100` with no `class_weight`, so it remains a probability
7. Categories at exactly Low <40, Medium 40–70, High >70
8. Same scenario seed produces identical scores across runs
9. `sector` absent from the design matrix; `payment_history_pattern` still absent
10. Scoring a too-small scenario raises an error naming found and required counts

**SHOULD**

11. `unscored_client_count` reported rather than defaulting missing clients to 0
12. `majority_baseline` reported alongside every evaluation, for context

**MUST NOT**

13. No `class_weight="balanced"` — it would break the score's meaning (D1)
14. No model persisted to disk (ADR-007)
15. No `sklearn` import outside `adapters/scoring/`
16. No hyperparameter tuning before the simple model is understood

## Open questions

  predates Finding 2.
- Retail's AUC (0.605–0.710) is materially weaker than manufacturing's (0.681–0.819). Expected —
  retail carries the highest `partial` weight and partial payers sit mid-gradient — but if the
  demo uses retail, that is the number the audience sees.
