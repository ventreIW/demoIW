# s4.4 Design — Score Explanation

**Owner:** Rodrigo · **Complexity:** Moderate (2 components, 0 new dependencies, custom ranking
and phrasing logic) · **Branch:** `story/s4.4/score-explanation`

## Problem

s4.3 produces a number. A collections officer cannot act on a number they do not understand, and
cannot defend a prioritisation they cannot explain to a client or a manager. RF-02.3 requires a
human-readable statement of the factors behind each score.

## Value

Every score arrives with a short Spanish sentence naming what actually drove it — grounded in the
model's own arithmetic, not a narrative invented alongside it.

## UX gate

This story produces user-facing prose, which triggers the design skill's UX gate. `/rai-research`
was **not** run: the audience (collections officers), the language (Spanish) and the medium (a
short sentence in a case list) are already fixed by the PRD and E5's design. What research would
resolve — *does this read well?* — is better answered by generating explanations on a real
scenario and reading them, which the plan does as its final task.

## Measurement (taken before designing)

Contributions from a real 600-client retail scenario, seed 42.

|---|---:|

| `pct_invoices_settled` | 0.1001 |
| `days_overdue_max` | 0.0622 |
| `avg_days_late_historical` | 0.0496 |
| `invoice_count` | 0.0440 |
| `has_partial_payments` | 0.0265 |
| `avg_invoice_amount` | 0.0140 |

| Rank | Median share | p25 |
|---|---:|---:|
| 1 | 1.000 | 1.000 |

| 4 | 0.361 | 0.257 |
| 5 | 0.261 | 0.170 |

### The grouping problem is real, not theoretical

`days_overdue_max` and `days_overdue_mean` contributions correlate at **0.885**, and **26.2% of
clients have both in their top tier**. Without grouping, more than a quarter of explanations
would state overdue ageing twice — which a collections officer reads as padding.

## Decisions

### D1 — Materiality: relative threshold at 30%, capped at 3 factors

Include any factor whose |contribution| is ≥ 30% of that client's largest. Cap at 3 so the
sentence stays readable.

Against the measured decay this yields 2–3 factors for a typical client and **1 when a factor
genuinely dominates** — which is what "no padding" has to mean in practice. A fixed top-3 rule
would present noise as reasons; an absolute threshold would misbehave for clients whose
contributions are all small.

### D2 — Correlated features are grouped before ranking

`days_overdue_max` and `days_overdue_mean` collapse into one **ageing** factor whose contribution
is the sum of both, phrased using `days_overdue_max` because "hasta 78 días" is a concrete fact an
officer can verify, while a mean is an abstraction.

Grouping happens **before** ranking, so the combined ageing signal competes fairly against
`outstanding_amount` and `pct_invoices_settled` rather than splitting its own weight — the same
problem that produced unstable coefficients in s4.3, appearing here as a presentation issue.

Only this pair is grouped. No broader grouping is invented without measurement behind it.

### D3 — Phrase the business fact, never the feature

The officer sees `"facturas con hasta 78 días de atraso"`, never `days_overdue_max` and never a
coefficient. Each factor owns a template that renders the *underlying value*, so the explanation
reads as a statement about the client rather than about the model.

### D4 — Deterministic template text, no LLM

Required, not merely preferred: `OPENROUTER_API_KEY` is still unavailable (E3's descoped gate
item, s4.8), and an LLM would make explanations non-reproducible — the same client could receive
different reasoning on two runs of the same seeded scenario.

## Gemba findings

| Finding | Consequence |
|---|---|
| `IScoringPort.feature_contributions` returns exact `coef × scaled value` per row | Consume directly; no new maths |
| `Score.explanation` already exists and is populated with a placeholder by `_placeholder_explanation` | **Replace it** — see legacy sweep |
| Frontend Spanish strings are terse and sentence-case (`"Sin datos"`, `"Cancelar selección"`) | Match that register — short, no exclamation, no marketing tone |
| No explanation or narrative-generation code exists anywhere | Net-new |
| `LLMEnrichmentService` exists but is for dataset enrichment | Not reused — different purpose, and unavailable without a key |

### Legacy sweep

**V1 is `_placeholder_explanation` in `app/application/use_cases/score_scenario.py:75`.** It is
superseded by `ScoreExplainer` and **deleted in this story** — not left beside the new path. A
grep for `_placeholder_explanation` must return nothing when this story closes.

### Drift risk

`governance/drift-hotspots.json` absent. **AG2 (clone amplification):** the ageing-grouping rule
must exist in exactly one place; if the explainer and any future UI both decided what to group,
they would diverge. Grouping lives in `ScoreExplainer` alone, and s4.5/E5 consume its output.

## Approach

| Component | Change | Purpose |
|---|---|---|
| `app/application/services/score_explainer.py` | create | Contributions → grouped, ranked, Spanish text |
| `app/application/use_cases/score_scenario.py` | modify | Use the explainer; **delete `_placeholder_explanation`** |
| `tests/unit/test_score_explainer.py` | create | Unit coverage |
| `tests/integration/test_score_explanation_integration.py` | create | Real scenarios, real prose |

**Lean gates.** KISS: pure functions over a contributions dict — no class hierarchy, no template
engine. DRY: one grouping rule, one phrasing table. YAGNI: no counterfactuals, no per-officer
tone, no i18n indirection (Spanish only, per project constraints; E7 owns translation). MVP:
one sentence per score.

**Supply chain:** N/A — no new dependency.

**Missing-data strategy:** a client whose contributions are all near zero still gets a sentence
(the score's headline plus its single largest factor). An empty `explanation` would violate the
`Score` contract and is asserted against.

## Examples

```python
from app.application.services.score_explainer import explain

contributions = {
    "days_overdue_max": -0.31,
    "days_overdue_mean": -0.24,      # grouped with the above
    "pct_invoices_settled": -0.28,
    "outstanding_amount": -0.11,
    "avg_days_late_historical": -0.04,
    "invoice_count": 0.02,
    "has_partial_payments": -0.02,
    "avg_invoice_amount": 0.01,
}
facts = {
    "days_overdue_max": 78.0,
    "pct_invoices_settled": 0.29,
    "invoice_count": 7.0,
    "outstanding_amount": 18_500.0,
    "has_partial_payments": True,
}

explain(score=34.2, contributions=contributions, facts=facts)
# 'Riesgo alto: facturas con hasta 78 días de atraso, sólo 2 de 7 facturas
#  liquidadas y $18,500 pendientes de cobro.'
```

```python
# One dominant factor — no padding
explain(score=22.0,
        contributions={"days_overdue_max": -0.62, "days_overdue_mean": -0.51,
                       "pct_invoices_settled": -0.09, "outstanding_amount": -0.05},
        facts={"days_overdue_max": 145.0, "pct_invoices_settled": 0.8,
               "invoice_count": 5.0, "outstanding_amount": 4_000.0})
# 'Riesgo alto: facturas con hasta 145 días de atraso.'
```

```python
# High score — positive drivers
explain(score=82.4,
        contributions={"pct_invoices_settled": 0.44, "days_overdue_max": 0.21,
                       "days_overdue_mean": 0.18, "outstanding_amount": -0.03},
        facts={"days_overdue_max": 0.0, "pct_invoices_settled": 1.0,
               "invoice_count": 6.0, "outstanding_amount": 2_100.0})
# 'Buen perfil de pago: 6 de 6 facturas liquidadas y sin facturas vencidas.'
```

## Acceptance criteria

**MUST**

1. Ageing appears as **one** factor — `days_overdue_max` and `days_overdue_mean` never both named
2. Factors below 30% of the client's largest contribution are omitted; at most 3 are named
3. A client with one dominant factor gets a one-factor explanation, not a padded three
4. Named factors match the score's direction (low score → predominantly negative drivers)
5. No feature names, coefficients, percentages-of-model or statistical vocabulary in the output
6. Same scenario seed → byte-identical explanations
7. `Score.explanation` is never empty
8. `_placeholder_explanation` is **deleted**; a grep for it returns nothing

**SHOULD**

9. Text matches the frontend's Spanish register — terse, sentence-case
10. Monetary values formatted readably (`$18,500`, not `18500.0`)

**MUST NOT**

11. No LLM call — must work without `OPENROUTER_API_KEY` and must be deterministic
12. No `sklearn` import outside `adapters/scoring/` — **asserted by a test**, since s4.3 wrote
    this as prose and violated it within the same story
13. No counterfactual claims ("pague X y su score sube") — a different story, and unverified

## Open questions
