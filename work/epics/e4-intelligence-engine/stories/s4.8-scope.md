# s4.8 Scope — Verify LLM Enrichment Against a Real Model

Status: **Proposed — BLOCKED on `OPENROUTER_API_KEY`.** Run `/rai-story-start` once a key exists.

## Why this story exists

This is E3's acceptance-gate item 3, **descoped at epic close on 2026-07-20 because no
OpenRouter API key is available to the team**:

> LLM enrichment runs (company names are not raw Faker names)

E3 delivered `OpenRouterAdapter`, `LLMEnrichmentService` (20 clients/batch, ADR-005) and the
prompt template `prompts/data_enrichment/v1_company_description.txt`. All of it is tested — but
only against `respx` mocks. **The qualitative half of E3 has never executed against a real
model.**

## The risk this closes

The failure is silent by construction:

- `OPENROUTER_API_KEY` defaults to `""` (`backend/app/config.py:11`)
- `LLMEnrichmentService` degrades gracefully when the call fails
- `test_generate_scenario_llm_failure_graceful_degradation` asserts the endpoint still
  returns **201**

With no key, the system returns success and raw Faker company names. Tests pass, the UI works,
and the demo looks correct until someone reads the names. A dead AI subsystem is currently
indistinguishable from a working one.

**Blast radius beyond E3:** OpenRouter also backs **RF-04** (E5 assisted communications) and
**RF-06.3-4** (E6 NL query). Three of the six system modules depend on this credential, against
a demo dated **2026-08-14**.

## In Scope

### 1. Obtain and configure the credential
- `backend/.env` with `OPENROUTER_API_KEY` and `MODEL_DATA_ENRICHMENT`
- Confirm `.env` is gitignored (it is) and that no key reaches the repository
- Prefer a cost-effective model per the project constraint

### 2. Verify enrichment end-to-end
- Generate a scenario against the real model
- Confirm company names differ from the Faker input and read as plausible Mexican companies
- Confirm `sector_description` is populated and sector-appropriate
- Confirm batching works across a batch boundary (>20 clients → more than one call)
- Record the observed output in the story retrospective — evidence, not assertion

### 3. Make degradation visible (the durable fix)
- Warn loudly at startup when `OPENROUTER_API_KEY` is empty
- Surface enrichment status in the generate response (e.g. `enriched: true|false`) so a
  degraded run is distinguishable from a successful one at the API and in the UI
- Keep graceful degradation — the endpoint should still succeed; it must simply stop
  *pretending* the AI ran

## Out of Scope

| Item | Reason |
|---|---|
| Prompt tuning / output-quality iteration | Verification first; quality work needs a baseline |
| Cost tracking or budget alerts | Parking lot if wanted |
| Retry/rate-limit changes to `OpenRouterAdapter` | Already delivered in s3.2 |
| E5 comms and E6 NL-query verification | Their own epics — but both unblock with this key |
