# s4.7 Design — i18n Completion (Switcher + s3.4 Retrofit)

**Story:** s4.7 | **Complexity:** Simple | **Status:** Draft

## Problem

Two i18n gaps remain after `b15-i18n-setup` landed (2026-07-08):

1. **No locale switcher UI** — the next-intl v4 foundation exists (routing, messages, `renderWithIntl`) but there's no user-facing control to toggle between ES and EN. Users must change browser language.
2. **GenerateScenarioForm ignores i18n** — shipped by s3.4 *after* `b15` landed, it contains ~27 hardcoded Spanish strings and a hardcoded `es-MX` date format. It's the only component that ignores the i18n foundation.
3. **Scenarios page also hardcoded** — `page.tsx` has 4 hardcoded Spanish strings: "Escenarios", "Selecciona un escenario para comenzar la demo.", "Error al cargar escenarios", and "Generar nuevo".

Additionally, the scenario-size minimum must be raised to 200 (from s4.3 findings that n<200 produces unreliable scores).

## Value

Demo facilitators can present in either Spanish or English from within the app, without browser configuration. The newest feature becomes the first fully i18n-compliant component, setting a pattern for all future UI work.

## Approach

Three independent changes, all in the frontend:

### 1. Locale Switcher (net-new)
- Create `src/components/locale/LocaleSwitcher.tsx` — a client component using next-intl's `useLocale()`, `useRouter()` (from `@/i18n/routing`, not `next/navigation`), and `usePathname()`
- When clicked, it calls `router.replace(pathname, { locale: nextLocale })` — next-intl handles the `as-needed` prefix logic automatically
- Place in `MainLayout` sidebar header, below the app title
- Shows "EN" when current locale is `es`, "ES" when current locale is `en`

### 2. GenerateScenarioForm Retrofit
- Import `useTranslations('scenarioForm')` and `useFormatter()` from `next-intl`
- Replace all hardcoded strings with `t('key')` lookups
- Replace `new Date(...).toLocaleDateString('es-MX')` with `formatter.dateTime(date, { dateStyle: 'long' })`
- Change `min={10}` → `min={200}` on the client-count input
- Update helper text key to reflect new minimum
- Sector labels move from hardcoded `SECTORS` array to message keys (re-index by sector value)

### 3. Messages
- Add `scenarioForm.*` and `scenariosPage.*` namespaces to both `es.json` and `en.json` (~24 keys total)
- Keep key sets identical between languages (no missing translations)

### 4. Scenarios Page Retrofit
- `page.tsx` is a **server component** — use `getTranslations()` from `next-intl/server` instead of `useTranslations()`
- Replace 4 hardcoded Spanish strings: title "Escenarios", description text, error fallback string, and the disabled "Generar nuevo" button text
- Pass locale from params (already available via `[locale]` route segment) or use `getRequestLocale()`

### Legacy Sweep
Nothing orphaned — this is net-new functionality (LocaleSwitcher) plus a retrofit of an existing component to use the established i18n foundation. The `renderWithIntl` utility remains valid; it will be extended with optional locale parameterization.

### Drift Risk
No drift risks identified. The changes are confined to 3 frontend modules with no coupling edges to other stories.

## Key Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Switcher uses next-intl `useRouter`, not `next/navigation` | next-intl handles `as-needed` locale prefix changes automatically (`/scenarios` ↔ `/en/scenarios`) |
| D2 | Switcher placed in MainLayout sidebar header | Visible on all pages, doesn't interfere with content, follows common SaaS pattern |
| D3 | `useFormatter()` for dates, not `toLocaleDateString` | Respects active locale without hardcoding |
|| D4 | Sector labels become i18n keys (not hardcoded enum) | Renders in active locale; sector values (API identifiers) stay as-is |
|| D5 | Scenarios page uses `getTranslations()` (server component) | `page.tsx` is async/server — can't use `useTranslations()`. `getTranslations()` from `next-intl/server` is the correct pattern for server components |

## Examples

### LocaleSwitcher usage
```tsx
// MainLayout.tsx — in sidebar, below app title:
<div className="border-b px-6 py-4">
  <div className="flex items-center justify-between">
    <span className="text-lg font-semibold text-slate-800">{t('app.title')}</span>
    <LocaleSwitcher />
  </div>
</div>

// LocaleSwitcher.tsx:
'use client'
import { useLocale } from 'next-intl'
import { useRouter, usePathname } from '@/i18n/routing'

export default function LocaleSwitcher() {
  const locale = useLocale()
  const router = useRouter()
  const pathname = usePathname()
  const next = locale === 'es' ? 'en' : 'es'

  return (
    <button
      onClick={() => router.replace(pathname, { locale: next })}
      className="text-xs font-medium text-slate-500 hover:text-slate-800"
    >
      {locale === 'es' ? 'EN' : 'ES'}
    </button>
  )
}
```

### GenerateScenarioForm i18n pattern
```tsx
// Before:
const SECTORS = [
  { value: 'manufacturing', label: 'Manufactura' },
  { value: 'retail', label: 'Retail' },
  { value: 'professional_services', label: 'Servicios Profesionales' },
]
// ... <label>Sector</label>
// ... min={10}
// ... <p className="text-xs text-slate-400">Mínimo 10, máximo 500</p>
// ... {new Date(scenario.created_at).toLocaleDateString('es-MX')}

// After:
const t = useTranslations('scenarioForm')
const formatter = useFormatter()

// ... <label>{t('sector')}</label>
// ... min={200}
// ... <p className="text-xs text-slate-400">{t('clientCountHelp', { min: 200, max: 500 })}</p>
// ... {formatter.dateTime(new Date(scenario.created_at), { dateStyle: 'long' })}

// Sector labels via t():
{value: 'manufacturing', label: t('sectors.manufacturing')}
{value: 'retail', label: t('sectors.retail')}
{value: 'professional_services', label: t('sectors.professional_services')}
```

### Scenarios page server component pattern
```tsx
// Before:
<h1 className="text-2xl font-bold text-slate-900">Escenarios</h1>
<p className="mt-1 text-sm text-slate-500">Selecciona un escenario para comenzar la demo.</p>
// ... error = e instanceof Error ? e.message : 'Error al cargar escenarios'
// ... <button disabled>Generar nuevo</button>

// After:
import { getTranslations } from 'next-intl/server'

export default async function ScenariosPage() {
  const t = await getTranslations('scenariosPage')

  return (
    <MainLayout>
      <div className="space-y-6">
        <div>
          <h1 className="text-2xl font-bold text-slate-900">{t('title')}</h1>
          <p className="mt-1 text-sm text-slate-500">{t('description')}</p>
        </div>
        {error ? (
          <div className="...">{t('errorLoading')}</div>
        ) : (
          <ScenarioGrid ... />
        )}
        <div className="flex gap-3">
          <button disabled className="...">{t('generateNew')}</button>
          <CsvUploadWrapper />
        </div>
      </div>
    </MainLayout>
  )
}

## Acceptance Criteria

| ID | Severity | Description | Verifiable |
|----|----------|-------------|------------|
| AC1 | MUST | Switcher visible in MainLayout; toggles ES↔EN without losing current page | Visual + test |
| AC2 | MUST | GenerateScenarioForm renders entirely from message keys — no hardcoded strings | Code review + test |
| AC3 | MUST | Client count input has `min=200`; helper text updated | Test assertion |
| AC4 | MUST | Dates format per active locale, not hardcoded `es-MX` | Test with locale change |
| AC5 | MUST | `es.json` and `en.json` key sets identical after changes | Script check |
| AC6 | MUST | Component tests use `renderWithIntl` and pass | `npm test` |
| AC7 | MUST | `npm run typecheck`, `npm run lint`, `npm test` pass | CI |
| AC8 | MUST NOT | No new hardcoded user-facing strings in modified files | Code review |
