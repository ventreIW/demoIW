# Story: Score Explanation

**Owner:** Rodrigo
**Branch:** `story/s4.4/score-explanation`

## User Story

As a **collections officer**,
I want each collectability score to come with a short, plain-language statement of the factors
that drove it,
so that I can judge whether to trust the number before acting on it, and explain my priorities to
a client or a manager.

## Acceptance Criteria

### Scenario: A low score explains itself

```gherkin
Given a client scored 34 with 78 days maximum overdue and 2 of 7 invoices settled
When the explanation is generated
Then it names the factors that lowered the score most
And it is written in Spanish, readable by a non-technical collections officer
And it contains no feature names, coefficients or statistical vocabulary
```

### Scenario: A high score explains itself

```gherkin
Given a client scored 82 with no overdue invoices and a strong settlement history
When the explanation is generated
Then it names the factors that raised the score most
```

### Scenario: Correlated features are not listed twice

```gherkin
Given days_overdue_max and days_overdue_mean both contribute strongly and similarly
When the explanation is generated
Then overdue ageing appears as a single factor, not as two separate ones
```

### Scenario: The explanation matches the score's direction

```gherkin
Given a client whose score is below the Low threshold
When the explanation is generated
Then the factors it names are predominantly negative contributors
```

### Scenario: Explanations are deterministic

```gherkin
Given a scenario generated with seed 42
When it is scored twice
Then every client receives an identical explanation both times
```

### Scenario: Only material factors are mentioned

```gherkin
Given a client where one factor dominates and the rest are negligible
When the explanation is generated
Then negligible factors are omitted rather than padded in to reach a fixed count
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|---|---|---|
| Score 34; `days_overdue_max=78`, `pct_invoices_settled=0.29`, partial payments | explain | "Riesgo alto: facturas con hasta 78 días de atraso, sólo 2 de 7 facturas liquidadas y pagos parciales pendientes." |
| Score 82; no overdue, 6/6 settled | explain | "Buen perfil de pago: sin facturas vencidas y 6 de 6 facturas liquidadas." |
| `days_overdue_max` and `days_overdue_mean` both strongly negative | explain | Ageing mentioned once, not twice |
| One dominant factor, others near zero | explain | Only the dominant factor is named |
| Same scenario, seed 42, explained twice | explain | Identical text |

## Notes

- Consumes `IScoringPort.feature_contributions`, which returns the exact linear decomposition
  `coef × scaled value` (ADR-007). No second library, no approximation.
- **Trustworthy only because of s4.3's fix.** Before ADR-007 Amendment 1, coefficient signs
  flipped between seeds in 19 of 27 configurations — an explanation built on that would have told
  an officer that being 78 days overdue *raised* the client's score. With `C=0.01`, signs are
  stable across all 45 measured configurations.
- **Correlated features must be grouped.** `days_overdue_max` and `days_overdue_mean` correlate at
  0.872, so they will contribute almost identically. Listing both would read as padding to a
  collections officer.
- Spanish UI language, per project constraints. The explanation is stored in the existing
  `Score.explanation` field, replacing s4.3's placeholder text.
- No LLM involvement. This is deterministic template text from numeric contributions — it must
  work without `OPENROUTER_API_KEY`, which remains unavailable.
