# Retrospective: S4.2 — Feature Engineering & Training Set

**Dates:** 2026-07-21 → 2026-07-21 (single session)
**Estimated:** S (5 tasks) | **Actual:** S — 5 tasks, no rework, no blockers

## Summary

Built the labelled training set for the collectability model: feature extraction from a scenario
snapshot, a forward-outcome labeller simulating 90-day collection, and a use case assembling both
into a `TrainingSet` split by client. 58 new tests; backend went 106 → 164, all gates clean.

The story's real output is not the code — it is **evidence that ADR-006's design works**. The
labeller reproduced the ADR's predicted collection rates within 0.002 across all six payment
patterns, so s4.3 starts from a validated assumption rather than a hopeful one.

## What went well

- **Sequencing T1 first paid off exactly as planned.** Moving the pattern profiles to the domain
  was guarded by 16 existing generator tests. Had the move altered behaviour, it would have
  failed before any new code depended on it. It didn't, and the safety net cost nothing.
- **The design's arithmetic held under measurement.** ADR-006 D2 predicted per-pattern collection
  probabilities from `Exponential(late_days_mean)`; the implementation reproduced every one to
  within 0.002. Writing the predictions down first turned the test from "does it run" into "is
  the model right".
- **Putting the riskiest task early worked.** T3 was the go/no-go. If the `on_time`-vs-`default`
  separation had been weak, ADR-006 would have needed revising before s4.3 — not after a model
  had been built on a meaningless label.
- **Gemba caught two things the story scope had wrong.** Partial payers emit a payment *smaller*
  than the invoice on an invoice still marked `overdue`, so outstanding must net them off; and
  `avg_days_late_historical` isn't stored at all — it has to be derived from
  `paid_date − due_date`. Both would have surfaced as bugs mid-implementation.
- **Tests that guard against passing trivially.** Determinism tests are paired with
  different-seed tests, and the class-balance test asserts a band rather than a point. A
  determinism test alone would pass on a constant.

## What to improve

- **The design under-specified the split's stratification.** It said "split by client" but not
  how to keep both classes on both sides when one class is small. Resolved during T4 by
  stratifying per label and never starving the train side, but it should have been a design
  decision rather than an implementation choice.
- **Two findings emerged only at T5, on real data.** `sector` being constant within a scenario,
  and the exclusion rate being 32–43%, were both visible in E3's code at design time. The gemba
  walk read the *generator's* logic but did not run it. **Reading code found the schema
  subtleties; only executing it found the distributional ones.**
- **A scripted cleanup deleted more than intended.** A `python3` heredoc meant to remove one
  duplicated constant removed both copies, breaking the module until caught by the next test run.
  A targeted `Edit` would have been safer than a regex over the whole file.

## Heutagogical Checkpoint

1. **What did I learn?**
   That leakage in synthetic data is structural, not accidental. Because `payment_history_pattern`
   *generates* every invoice and payment, any label derived from it makes the model an inverse of
   the generator. The fix isn't dropping a column — it's putting a time boundary between features
   and label. Also, concretely: `pandas.fillna` on object-dtype columns silently downcasts and is
   deprecated; cast first, then fill.

2. **What would I change about the process?**
   Run the generator during the gemba walk, not just read it. Both T5 findings — constant
   `sector`, 32–43% exclusions — were properties of the *output distribution*, invisible to code
   reading but obvious in one execution. A five-minute "generate 400 clients and describe the
   frame" step in story-design would have moved both findings from implementation to design.

3. **Framework improvements?**
   - `rai-story-design`'s gemba step says "read the modules". For data-processing stories it
     should also say **run them and inspect the output** — reading finds schema, executing finds
     distribution.
   - `rai-story-implement` Step 2.0 requires confirming every file edit interactively. For a
     5-task TDD story that is roughly 20 confirmations; the guidance would be better scoped to
     modifications of *existing* files, where the risk actually lives.
   - No `.raise/rai/learnings/` directory exists, so the chain-read step in review has nothing to
     aggregate. Either the CLI should create it or the skills should stop assuming it.

4. **More capable of now?**
   Designing a leakage-free training set from causally-generated synthetic data, and — more
   transferable — writing an ADR whose numeric predictions can be checked against measured output
   in the same story. The pattern of "predict in the ADR, assert the prediction in the test"
   converts a design decision into a falsifiable claim.

## Findings carried to s4.3

1. **`sector` carries no signal.** Constant within a scenario, so its one-hot column has zero
   variance. Harmless under a linear model (centred value always 0, so it cannot pollute s4.4
   explanations) but dead weight. Decide: drop it, or keep for cross-scenario training.
2. **A user can generate a scenario that cannot be scored.** 32–43% of clients are excluded for
   having nothing outstanding, and `MIN_CLIENTS = 20` means scenarios under ~35 clients raise
   `InsufficientTrainingDataError`. The generation form allows 10. This is a product hole, not
   just a modelling detail.
3. **Use ROC-AUC and balanced accuracy, not raw accuracy.** Measured positive rate is 0.75–0.86
   by sector, so majority-class already scores that. Confirmed empirically, per ADR-006 D3.
