## Findings for s4.3

Neither was anticipated in the design. Both need a decision before the model is trained.

### 1. `sector` is constant within a scenario — the one-hot column carries no signal

`GenerationParams` takes a single sector for the whole scenario, and the generator writes it to
every client. So `pd.get_dummies` produces exactly one column (`sector_retail`), identical for
every row.

A zero-variance feature contributes nothing. Under ADR-007's linear model it will receive an
arbitrary coefficient, and since its centred value is always 0 it will never appear in an s4.4
explanation — so it is harmless but dead. It would only carry signal if training spanned
scenarios of different sectors.

Options: drop `sector` from the design matrix; keep it for cross-scenario training later; or
make sector a per-client attribute (a change to E3's generator, larger).

### 2. 32–43% of clients are excluded for having nothing outstanding

Expected per ADR-006 D4, but the magnitude was not: a 400-client scenario yields ~230–270
training rows. With `MIN_CLIENTS = 20`, scenarios below roughly 35 clients will raise
`InsufficientTrainingDataError`.

The generation form allows 10–500 clients, so **a user can generate a valid scenario that cannot
be scored.** s4.3 needs to decide what the API does then — a clear error, or a minimum enforced
at generation time.
