# Retrospective: S4.5-API — Prioritized List Endpoint

**Dates:** 2026-07-22 (single session)
**Estimated:** S (7 tasks) | **Actual:** S — 7 tasks, no rework, no blockers

## Summary

Exposed the `PrioritizeScenario` use case as `GET /api/v1/scenarios/{id}/prioritized` with query params for threshold, sort, category filter, and days_overdue_min. 12 tests; backend 308 → 308 (no new backend logic tests needed, endpoint is thin).

The endpoint is a contract boundary: it serializes exactly the `PrioritizedCase` / `PrioritizedPortfolio` shapes from s4.5-formula (commit 5d8980f). Any change there is a contract change.

## What went well

- **Contract-first design worked.** The endpoint was built against the value objects from `5d8980f` (T1). No rework on the contract; the only schema decision was whether to include `days_overdue` on `PrioritizedCase` — deferred because the field doesn't exist on the domain object yet.
- **Validation at the edge caught 3 classes of bad params.** Invalid sort field, invalid category, invalid threshold all return 422 with clear messages. The 422 tests are explicit and will fail if someone silently removes validation.
- **Reusing the mock LLM pattern from `test_openrouter_adapter.py` made integration tests deterministic.** 100 clients, seed 42, mocked enrichment → full end-to-end path exercises scoring → prioritization → serialization.
- **12 tests cover the full surface.** Happy path, threshold, sort (3 fields × 2 orders), category filter (High/Low), edge cases (empty portfolio, nonexistent scenario, 4 invalid params). 0 flaky tests.
- **No new backend logic tests needed.** The prioritizer and portfolio logic were already tested in s4.5-formula (48 tests). This story only tested the HTTP layer — routing, serialization, validation.

## What to improve

- **`days_overdue_min` is a documented stub.** The PrioritizedCase domain object doesn't have `days_overdue`; the filter silently passes all cases through. This is by design (follow-up for E5 workload policy), but the test documents the current behavior. When the field is added to the domain, the test will verify it without modification.
- **Re-scoring on every request is a known demo limitation.** Scores aren't persisted yet (s4.9), so the endpoint re-runs `ScoreScenario` on each call. Acceptable for demo; will be removed when s4.9 lands.
- **Category filter is case-sensitive string match.** If the frontend sends "high" vs "High", it filters to zero. Could normalize at the API boundary, but the contract uses `ScoreCategory` enum values — the frontend should send exactly those.

## Heutagogical Checkpoint

1. **What did I learn?**
   - A thin HTTP wrapper over a tested domain use case is fast to build and easy to verify — the real work was already done in s4.5-formula.
   - Query parameter validation at the router level (before any DB/Scoring calls) is cheap insurance. The 422 tests are shorter than the happy path and more likely to catch regressions from schema drift.
   - Contract boundaries are the *only* thing that should be in the API layer. The endpoint has no business logic — it validates, calls `PrioritizeScenario`, serializes. That separation makes both layers easy to test independently.

2. **What would I change about the process?**
   - Make the contract review explicit before starting the API story. The endpoint was blocked on s4.5-formula T1 (contract in 5d8980f). If that review had been a formal step, we'd have caught the `days_overdue` question earlier and either added it to the domain or documented the deferral in the scope.

3. **Framework improvements?**
   - The design skill's "contract boundary" section could explicitly list: value object shapes, units, derived properties, and which fields are deferred.
   - A repo-level check for "unused public API" (AC #13 from s4.5-formula) would be stronger than per-story manual checks. Three stories shipped unused APIs before this pattern was established.

4. **More capable of now?**
   - Building thin FastAPI endpoints over existing domain logic with confidence. The pattern — validate params → call use case → serialize domain objects → return — is now reflexive.
