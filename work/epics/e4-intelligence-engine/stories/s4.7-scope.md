# s4.7 Scope — i18n Completion (Switcher + s3.4 Retrofit)

Status: **Proposed** — E4 is not yet authorized. Run `/rai-story-start` when it is.

## Why this story exists

E4's original s4.1 ("i18n foundation") was delivered out-of-band by backlog item
`b15-i18n-setup`, merged 2026-07-08: next-intl v4, `[locale]` routing, middleware locale
detection, `messages/{es,en}.json`, ICU pluralization, dynamic `html lang`, and the
`renderWithIntl` test wrapper.

Two gaps remain. One was scoped out of `b15` deliberately; the other is a regression.

## In Scope

### 0. Raise the scenario-size minimum (added 2026-07-21, from s4.3)

- `GenerateScenarioForm.tsx`: change the client-count input from `min={10}` to **`min={200}`**
- Update the helper text ("Mínimo 10, máximo 500" → "Mínimo 200, máximo 500")
- Add the reason to the field's help text: scenarios below 200 clients cannot be scored reliably

**Why 200.** s4.3 measured model quality across five seeds, three sectors and four scenario
sizes. At n=100 the ROC-AUC ranges **0.286 to 1.000** — worse than chance to perfect — because
~100 clients yields roughly 60 scoreable and an ~18-row test set. Quality only stabilises from
n≥200 (minimum observed 0.559, mean 0.698).

Today the form allows 10 clients while scoring needs at least 20 with an outstanding balance, and
32–43% of clients are excluded for having nothing outstanding. **A user can currently generate a
scenario that cannot be scored at all** — and, worse, one that produces authoritative-looking
scores from meaningless data.

The backend deliberately stays permissive so E3's fast integration tests (3–5 clients) keep
working; s4.3 owns the scoring-side error. This form change is the user-facing guard.

### 1. Locale switcher UI
- User-facing control to switch between `es` and `en` from within the application
- Placed in `MainLayout` (already uses `useTranslations()`)
- Navigates to the equivalent route in the target locale, preserving the current path
  (`/scenarios` ↔ `/en/scenarios`) — use next-intl's `Link`/`useRouter` from `@/i18n/routing`
- Reflects the active locale in its own label
- `b15` listed this under "Out of Scope"; it is E9's only legitimate story (B-18)

### 2. `GenerateScenarioForm.tsx` retrofit (regression)
- Replace **27 hardcoded Spanish strings** with `useTranslations()` keys
- Replace hardcoded `toLocaleDateString('es-MX')` with locale-aware formatting
  (next-intl `useFormatter`)
- Add the corresponding keys to **both** `messages/es.json` and `messages/en.json`
  (currently 18 keys each, all from `b15`)
- Shipped by s3.4 *after* `b15` landed, making the newest feature the only one that ignores
  the i18n foundation

## Out of Scope

| Item | Reason |
|---|---|
| Full EN copy review / translation pass | E7 (may be dropped if the demo is Spanish-only) |
| Persisting locale preference to the database | `b15` decision: browser-only for now |
| RTL support | Not needed for ES/EN |
| Backend i18n | All endpoints return structured data, no UI strings |
| Locale detection from user profile | No user profile exists yet |

## Done when

- [ ] Switcher visible in `MainLayout` and changes locale without losing the current page
- [ ] `GenerateScenarioForm` renders entirely from message keys — no hardcoded Spanish
- [ ] Dates in `GenerateScenarioForm` format per active locale, not fixed `es-MX`
- [ ] `es.json` and `en.json` have identical key sets (no missing translations either way)
- [ ] Component tests use `renderWithIntl` (pattern PAT-R-3 from `b15`)
- [ ] A test asserts the switcher actually changes rendered language
- [ ] `npm run typecheck`, `npm run lint`, and `npm test` pass
- [ ] No new hardcoded user-facing strings anywhere in `src/`

## Notes

- Follow patterns **PAT-R-2** (next-intl v4 setup) and **PAT-R-3** (i18n test wrapper),
  both captured in the `b15` retrospective.
- `b15`'s retrospective warns that any component tree containing `useTranslations()` needs
  `NextIntlClientProvider` in tests — *including indirect callers*. `ScenarioGrid` hit this.
- Guard against recurrence: this regression happened because a story shipped UI after the
  i18n foundation landed without picking it up. Consider a lint rule or review checklist item
  for hardcoded user-facing strings.
