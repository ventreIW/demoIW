# s4.2 Design — Feature Engineering & Training Set

**Owner:** Rodrigo · **Complexity:** Moderate (4 components, 0 external integrations, custom
labelling algorithm) · **Branch:** `story/s4.2/feature-engineering`

## Problem

E4 needs a labelled training set, and the obvious way to build one is a trap. Every invoice and
payment in the dataset is generated from the client's `payment_history_pattern`, which is stored
on the client record. Feed it to a model and accuracy goes to ~100% by reading the answer key.
There is also no natural outcome to predict: the generator emits a snapshot, not a resolved
history.

## Value

A training set where features and label are separated by a time boundary, so the model learns to
predict collection behaviour instead of inverting the generator. Without this, every downstream
story (s4.3, s4.4, s4.5) is built on a number that means nothing.

## Gemba findings

Read before designing.

| Finding | Consequence |
|---|---|
| `clients` columns: `id, name, sector, payment_history_pattern` | The leaking column is right there. Exclusion must be enforced, not remembered |
| `invoices` columns: `id, client_id, folio, amount, issue_date, due_date, days_overdue, status` | `status ∈ {overdue, paid}` is the settled/open discriminator |
| `payments` columns: `id, invoice_id, amount, paid_date` | Historical lateness = `paid_date − due_date`, joined through `invoice_id`. Not stored directly — must be derived |
| Partial payers emit a payment **smaller than** the invoice, on an invoice still `overdue` | **`outstanding_amount` must net off partial payments.** Summing invoice amounts would overstate exposure for exactly the cohort that matters |
| `_PatternProfile` / `_PATTERN_PROFILES` are private, inside the adapter | Moving to domain — see decision D1 |
| `GenerateDataset` instantiates `ProceduralGenerator` directly rather than through `IDatasetPort` | Pre-existing architecture smell. **Not fixed here** — out of scope, noted for the parking lot |
| No feature, labelling, or training-set code exists anywhere | Net-new |

### Legacy sweep

**Nothing becomes orphaned — this is net-new.** No V1 of feature extraction, labelling or
training-set construction exists. The one modification to existing code is D1, which *moves*
the pattern profiles rather than superseding them: the generator keeps using the same values
from a new location, and no duplicate definition is left behind.

### Drift risk

`governance/drift-hotspots.json` is absent — hotspot ranking skipped.

- **AG2 (clone amplification):** the live risk. Copying `_PATTERN_PROFILES` into the labeller
  would create two definitions of the causal model that drift apart silently. D1 removes the
  risk structurally rather than relying on discipline.
- **AG4 (context-window fan-out):** not a concern. Four new files, one modified, one module.

## Decisions

### D1 — Pattern profiles move to the domain layer

`_PatternProfile` and `_PATTERN_PROFILES` move from
`app/adapters/dataset/procedural_generator.py` to
`app/domain/value_objects/payment_behaviour.py` as public `PaymentBehaviourProfile` and
`PATTERN_PROFILES`. The generator and the labeller both import from there.

**Rationale.** These profiles encode how each client archetype behaves — domain knowledge that
leaked into an adapter. The labeller lives in `application/services/`; importing an adapter's
private names would invert the hexagonal dependency and ignore the underscore that says "do not
import me". Moving is a smaller lie than either alternative, and it gives one source of truth
for the causal model as ADR-006 requires.

Covered by the 16 existing generator tests: if the move breaks anything, they fail.

### D2 — Collection delay is drawn from the client's own profile

For each client, draw `days_to_collect ~ Exponential(late_days_mean)` using the scenario's
seeded RNG. Label is `1` when `days_to_collect <= 90`.

This produces a monotone gradient that matches domain intuition:

| Pattern | `late_days_mean` | P(collect ≤ 90d) |
|---|---:|---:|
| `on_time` | 3 | ~1.00 |
| `delayed_30` | 30 | ~0.95 |
| `delayed_60` | 60 | ~0.78 |
| `delayed_90_plus` | 100 | ~0.59 |
| `partial` | 50 | ~0.83 |
| `default` | 160 | ~0.43 |

### D3 — Expect class imbalance, and do not measure with raw accuracy

Weighting those probabilities by the retail sector mix gives roughly **80% positive class**. A
majority-class baseline therefore scores ~80% accuracy, which would make a useless model look
good.

**s4.3 must evaluate on ROC-AUC and balanced accuracy**, with majority-class as the stated
baseline. Recorded here because the choice is forced by the label design, not by s4.3.

### D4 — Clients with no outstanding balance are excluded from training

A client whose invoices are all settled has nothing to collect, so "will it be collected" is
undefined. They are dropped from the training set with a reported count — not silently, and not
labelled `1` by default.

## Approach

Four new components plus one move. Each does one thing.

| Component | Change | Purpose |
|---|---|---|
| `app/domain/value_objects/payment_behaviour.py` | create | Public `PaymentBehaviourProfile` + `PATTERN_PROFILES` (moved, D1) |
| `app/adapters/dataset/procedural_generator.py` | modify | Import profiles from domain; definitions removed |
| `app/domain/value_objects/client_features.py` | create | Feature row schema + the canonical `FEATURE_COLUMNS` list |
| `app/application/services/feature_extractor.py` | create | `RawDataset` → one feature row per client |
| `app/application/services/outcome_labeller.py` | create | Seeded 90-day forward simulation per client |
| `app/domain/value_objects/training_set.py` | create | `X`, `y`, `client_ids`, split metadata |
| `app/application/use_cases/build_training_set.py` | create | Wires extraction + labelling + split |

**Lean gates.** KISS: pandas groupby aggregations, no framework. DRY: D1 removes the only
duplication risk. YAGNI: no feature scaling (belongs with the model), no feature-selection
experiments, no persistence — the training set is built on demand per ADR-007. MVP: this story
ends with a `TrainingSet` in hand; nothing trains yet.

**Supply chain gate:** N/A. No new dependency — `numpy` and `pandas` are already present.
`scikit-learn` arrives in s4.3 under ADR-007.

**Missing-data strategy** (declared explicitly rather than discovered):

| Case | Behaviour |
|---|---|
| Client with no invoices at all | Excluded from the training set, counted and reported |
| Client with no *open* invoices | Excluded per D4, counted and reported |
| Client with no settled invoices | `avg_days_late_historical = 0.0`, `pct_invoices_settled = 0.0` — zeros, never NaN |
| Invoice with no payment rows | Outstanding = full amount |
| Only one label class present | Raises `InsufficientTrainingDataError` naming the problem |

### Building the training set

```python
from app.application.use_cases.build_training_set import BuildTrainingSet

ts = BuildTrainingSet().execute(raw_dataset, seed=42, horizon_days=90, test_fraction=0.3)

ts.X_train.shape          # (70, 11)   one-hot sector expands the 9 base features
ts.y_train.mean()         # ~0.80      imbalance is expected — see D3
set(ts.train_client_ids) & set(ts.test_client_ids)   # set()  — disjoint by construction
ts.excluded_client_count  # 12         no outstanding balance, D4
```

### Error path

```python
ts = BuildTrainingSet().execute(tiny_uniform_dataset, seed=42)
# InsufficientTrainingDataError:
#   training set has a single label class (all collected_within_90d=1);
#   need both classes to train. 8 clients, 12 excluded for no outstanding balance.
```

## Acceptance criteria

Refines the Gherkin scenarios in `s4.2-story.md` — not a duplicate of them.

**MUST**

1. `payment_history_pattern` is absent from the feature frame, asserted by a test that fails
   loudly if reintroduced
2. Same scenario seed produces byte-identical features **and** labels across runs
3. `outstanding_amount` nets off partial payments — a partial payer's outstanding is strictly
   less than the sum of their open invoice amounts
4. Split is by client: `train ∩ test = ∅`, and both label classes appear on each side
5. `on_time` clients show a materially higher collection rate than `default` clients over a
   large sample
6. Single-class training set raises `InsufficientTrainingDataError` naming the cause
7. Clients with no settled invoices yield `0.0`, never `NaN`
8. Generator behaviour is unchanged after D1 — the 16 existing generator tests still pass

**SHOULD**

9. Excluded-client count is reported on the `TrainingSet`, not silently dropped
10. `FEATURE_COLUMNS` is a single named constant, so s4.3 and s4.4 cannot drift from it

**MUST NOT**

11. No model training, scoring or evaluation — that is s4.3
12. No duplicate copy of the pattern profiles anywhere in the tree
13. No HTTP endpoint — nothing consumes this yet
