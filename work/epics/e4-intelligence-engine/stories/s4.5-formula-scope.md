# s4.5-formula Scope — Priority Value & Pareto Threshold

**Owner:** Rodrigo · **Size:** S · **Branch:** `story/s4.5/priority-formula`
**Hands off to:** s4.5-API (Nano)

## In Scope

- `PrioritizedCase` value object — client, score, outstanding, expected recoverable, rank
- Priority value: **expected recoverable value = outstanding × score/100**, in pesos
- Ranking over a scored portfolio, with deterministic tie-breaking
- Pareto filter: smallest prefix of the ordering holding ≥80% of total expected recoverable value
- Concentration reporting — how few clients hold how much value, so the leverage is visible
- `PrioritizeScenario` use case consuming a `ScoringRun`
- Degenerate-portfolio handling (single dominant client, all-equal values)
- **The contract handed to s4.5-API**, documented well enough for Nano to build against without
  re-deriving the formula

## Out of Scope

| Item | Reason |
|---|---|
| Sortable / filterable list endpoint (RF-03.3–4) | **s4.5-API (Nano)** — this story defines the formula it serves |
| Operations panel UI | E5 s5.1 |
| Persisting priorities | Derived on read from scores; no storage decision needed |
| Rescore-on-contact | s4.6 (Nano) |
| Collector capacity modelling / workload balancing | Genuinely useful, genuinely a different problem. Parking lot |
| Time-discounting of expected value | Would need a collection-cost model that does not exist |

## Done when

- [ ] Expected recoverable value = `outstanding × score/100`, expressed in pesos
- [ ] A low-score/high-amount account can outrank a high-score/low-amount one, and does
- [ ] A 95-score account owing $200 does not lead the queue
- [ ] Pareto filter returns the **smallest** prefix reaching the 80% threshold
- [ ] The returned subset's expected value is ≥80% of the portfolio total
- [ ] Concentration is reported (subset size vs portfolio size, share of value)
- [ ] Single-dominant-client portfolio returns that client alone, without error
- [ ] All-equal portfolio does not return an empty subset
- [ ] Same seed produces identical values **and** identical ranks
- [ ] Ties break deterministically, asserted across repeated runs
- [ ] Clients without an outstanding balance are absent, not ranked last
- [ ] `pytest` green; `ruff check`, `ruff format --check`, `mypy app/` clean
- [ ] **No unused public API or parameter** — three stories running have introduced one; the AR
      gate caught each. Checked before close this time, not at it

## Risks

| Risk | Mitigation |
|---|---|
| The formula is defensible arithmetic but produces a queue that looks wrong to a collections manager | Print the top 20 of a real portfolio and read it, as s4.4 did — the ordering is the product, not the equation |
| 80% is adopted because "Pareto" says so | The threshold is a parameter; measure the actual concentration curve and check 80% sits somewhere meaningful on it rather than assuming |
| Expected-value ranking buries small accounts permanently | Real concern for a collections team, out of scope here — belongs to workload policy in E5. Note it rather than silently encode it |
| Contract drift between this story and s4.5-API | The value object and its units are the handoff; document them in the design so Nano builds against a fixed shape |
