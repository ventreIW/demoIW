# Story: Priority Value & Pareto Threshold

**Owner:** Rodrigo · **Branch:** `story/s4.5/priority-formula`
**Hands off to:** s4.5-API (Nano) once the formula is agreed

## User Story

As a **collections manager with more accounts than collector-hours**,
I want the portfolio ordered by how much money each account is actually likely to recover, and the
subset holding most of that value identified,
so that a limited team works the accounts that matter instead of the accounts that shout loudest.

## Acceptance Criteria

### Scenario: Priority combines probability and amount

```gherkin
Given a client scored 30 owing $100,000
And a client scored 90 owing $30,000
When priority values are computed
Then the first client ranks above the second
And the ordering reflects expected recoverable value, not score or amount alone
```

### Scenario: A high score on a trivial balance does not lead the queue

```gherkin
Given a client scored 95 owing $200
When priority values are computed
Then that client does not appear above accounts holding materially more expected value
```

### Scenario: The Pareto subset concentrates the value

```gherkin
Given a scored portfolio of 300 clients
When the Pareto filter is applied at 80%
Then the returned subset holds at least 80% of the portfolio's total expected recoverable value
And it is the smallest prefix of the priority ordering that does so
```

### Scenario: The subset is materially smaller than the portfolio

```gherkin
Given a scored portfolio with a realistic spread of amounts
When the Pareto filter is applied at 80%
Then the subset contains fewer clients than the full portfolio
And the reduction is reported so an operator can see the leverage
```

### Scenario: Priority is reproducible

```gherkin
Given a scenario generated with seed 42
When priorities are computed twice
Then every client receives an identical priority value and rank
```

### Scenario: Ties are ordered deterministically

```gherkin
Given two clients with identical expected recoverable value
When priority values are computed
Then their relative order is stable across runs
```

### Scenario: A degenerate portfolio does not break the filter

```gherkin
Given a portfolio where one client holds nearly all outstanding value
When the Pareto filter is applied
Then it returns that client alone rather than failing or returning everyone
```

## Examples (Specification by Example)

| Client | Score | Outstanding | Expected recoverable | Rank |
|---|---:|---:|---:|---:|
| A | 30 | $100,000 | $30,000 | 1 |
| B | 90 | $30,000 | $27,000 | 2 |
| C | 60 | $20,000 | $12,000 | 3 |
| D | 95 | $200 | $190 | 4 |

| Input | Action | Expected output |
|---|---|---|
| Portfolio above, 80% threshold | Pareto filter | A + B (57k of 69.19k = 82%) |
| Two clients, identical expected value | rank | Stable order across repeated runs |
| One client holding 99% of value | Pareto filter | That client alone |
| Same scenario, seed 42, twice | compute | Identical values and ranks |

## Notes

- **This story is the formula only.** The sortable/filterable API is s4.5-API (Nano), and it
  starts once the formula here is agreed — so the contract this story produces is a handoff, not
  an internal detail.
- `score / 100` is a genuine probability (s4.3 D1 kept `predict_proba` calibrated with no
  `class_weight`), so **expected recoverable value = outstanding × score/100 has real units:
  pesos**. That is what makes the Pareto claim meaningful rather than an index-weighted heuristic.
- `outstanding_amount` already nets partial payments (s4.2 gemba finding), so the exposure figure
  is the true balance rather than the invoiced total.
- Pareto here is applied to **expected** recoverable value, not raw outstanding. Ranking by raw
  amount would push uncollectable large balances to the top — precisely the behaviour the
  scoring engine exists to correct.
- Clients with no outstanding balance have no score (ADR-006 D4) and are absent from the
  prioritized list, not ranked last.
