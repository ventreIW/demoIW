# s4.6 Plan — Rescore-on-Contact Endpoint

**Story ID:** s4.6 · **Epic:** E4 · **Size:** S (3-5 SP) · **Owner:** Nano
**Branch:** `story/s4.6/rescore-on-contact` · **Date:** 2026-07-22

## Overview

Expose `POST /api/v1/scenarios/{scenario_id}/clients/{client_id}/rescore` endpoint that takes a contact result, adjusts the client's score, and returns the updated prioritized portfolio with recalculated Pareto subset. Re-uses `PrioritizeScenario` use case and response models from s4.5-API.

### Task 2: Create RescoreScenario use case

**Description:** Create `RescoreScenario` class in `backend/app/application/use_cases/rescore_scenario.py`. Takes `scenario_id`, `client_id`, `contact_result`. Fetches scenario data, adjusts score via heuristics, calls `PrioritizeScenario` to re-rank, returns `PrioritizedPortfolio`.

**Score heuristics:**
- `promise_to_pay` → +10 (clamped to 100)
- `refusal` → -15 (clamped to 0)
- `partial_payment` → +5
- `no_answer`, `wrong_number` → no change

**TDD:**

- GREEN: Implement use case
- REFACTOR: Add error handling for unknown client

**AC ref:** "Rescore endpoint changes a score given a contact result", "Score clamping at 0/100"

## Execution Order Rationale

1. **Task 1 (endpoint)** — highest risk: new router code, parameter handling, error responses
2. **Task 2 (use case)** — core logic, must work before endpoint integrates it
3. **Task 3 (container)** — dependency wiring, must work before tests pass
4. **Task 4 (happy path)** — proves end-to-end works
5. **Task 5 (validation)** — defensive, lower risk
6. **Task 6 (edge cases)** — defensive, lower risk
7. **Task 7 (manual)** — final validation with running system

**Parallelizable:** Tasks 5 & 6 after 4 completes.

---

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Score adjustment heuristic is arbitrary | Demo uses simple heuristic; document as placeholder for real calibration |
| Re-scoring on every request is slow | Acceptable for demo (≤1000 clients); s4.9 will persist scores |
| Contact result enum diverges from domain | Use existing `ContactResultType` enum from domain/enums.py |
| `days_overdue` field missing on `PrioritizedCase` | Filter silently passes (returns all); document as known limitation |
