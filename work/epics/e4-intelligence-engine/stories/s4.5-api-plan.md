# s4.5-API Plan — Prioritized List Endpoint

**Story ID:** s4.5-API · **Epic:** E4 · **Size:** S (3-5 SP) · **Owner:** Nano
**Branch:** `story/s4.5/prioritized-list-endpoint` · **Date:** 2026-07-22

## Overview

Expose the `PrioritizeScenario` use case as `GET /api/v1/scenarios/{scenario_id}/prioritized` with query parameters for threshold, sorting, and filtering. The contract (`PrioritizedCase` / `PrioritizedPortfolio`) is defined in s4.5-formula (commit 5d8980f) — this story only wires it to HTTP.

### Task 1: Add endpoint to scenarios router

**Description:** Add `GET /api/v1/scenarios/{scenario_id}/prioritized` to `scenarios.py`. Accepts query parameters:
- `threshold: float = 0.80` (Pareto threshold override)
- `sort: str = "rank"` — one of `rank`, `score`, `outstanding`, `expected_recoverable`, `days_overdue`
- `order: str = "asc"` — `asc` | `desc`
- `category: str | None = None` — filter by `High` | `Medium` | `Low`
- `days_overdue_min: int | None = None`

Returns `PrioritizedPortfolio` shape. 404 if scenario has no scores.

### Task 2: Wire PrioritizeScenario in container

**Description:** Add `get_prioritize_scenario_use_case` provider to `container.py` that injects dependencies (same as `ScoreScenario` needs: scoring port, training set builder, etc.) — but actually `PrioritizeScenario` only needs a `ScoringRun`. The endpoint will:
1. Fetch the scenario
2. Re-run scoring (or fetch cached scores — but s4.9 not done yet)
3. Call `PrioritizeScenario().execute(scoring_run, threshold)`

Since scores aren't persisted yet (s4.9), the endpoint re-scores on each call. This is acceptable for demo.

## Execution Order Rationale

1. **Task 1** (endpoint) — highest risk: new router code, parameter handling, error responses
2. **Task 2** (container) — dependency wiring, must work before tests pass
3. **Task 3** (models) — shapes the response, needed for tests
4. **Task 4** (integration) — proves the happy path end-to-end
5. **Task 5** (filters/sorts) — each is independent, can parallelize
6. **Task 6** (edge cases) — defensive, lower risk
7. **Task 7** (manual) — final validation with running system

**Parallelizable:** Tasks 5 & 6 after 4 completes.

---

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `days_overdue_min` needs `days_overdue` on `PrioritizedCase` | Check domain object first; if missing, add derived property or note as follow-up |
| Re-scoring on every request is slow | Acceptable for demo; s4.9 will persist scores. Document as known limitation. |
| Contract drift from s4.5-formula | Build response models directly from value objects; any change = test failure |
| Sorting subset instead of full portfolio | Test explicitly asserts full portfolio sorted, subset is prefix |
| Category filter changes Pareto meaning | Document in OpenAPI: Pareto recomputed on filtered set |
