# s4.6 Scope — Rescore-on-Contact Endpoint

**Owner:** Nano · **Size:** S · **Branch:** `story/s4.6/rescore-on-contact`
**Prerequisite:** s4.3 merged (✅ done)

## In Scope

- `POST /api/v1/scenarios/{scenario_id}/clients/{client_id}/rescore` endpoint
- Request body: `{ "contact_result": "promise_to_pay" }` (enum of valid results)
- Response: Updated client score with new prioritization
- Contact result enum: `promise_to_pay`, `partial_payment`, `refusal`, `no_answer`, `wrong_number`
- Re-ranking logic: adjusts score based on contact result, re-ranks portfolio
- Uses existing `PrioritizeScenario` use case (s4.5-API) for re-ranking
- 404 for unknown scenario, unknown client, unscored scenario

## Out of Scope

| Item | Reason |
|------|--------|
| Score persistence (`IScoreRepository`) | s4.9 (separate story) |
| Full model re-fit | Not needed — rescore is fast re-rank |
| Collector UI consuming the endpoint | E5 s5.3 |
| Contact result validation beyond enum | Enum validation is sufficient |
| Batch rescore | Not needed for demo |

## Done when

- [ ] `POST /api/v1/scenarios/{scenario_id}/clients/{client_id}/rescore` returns 200 with updated score
- [ ] Contact result enum validation returns 422 for invalid values
- [ ] 404 for unknown scenario / unknown client / unscored scenario
- [ ] Rescore changes score based on contact result (promise_to_pay increases, refusal decreases)
- [ ] `pytest` green; `ruff check`, `ruff format --check`, `mypy app/` clean
- [ ] No unused public API or parameter (AR gate)

## Risks

| Risk | Mitigation |
|------|------------|
| Rescore logic is arbitrary without real calibration | Demo uses simple heuristic: +10 for promise_to_pay, -15 for refusal, 0 for others |
| Re-scoring every contact is slow | Demo scale (≤1000 clients) is fine; s4.9 will persist scores for production |
| Score can go outside 0–100 bounds | Clamp to [0, 100] after adjustment |
