# s4.5-formula Design — Priority Value & Pareto Threshold

**Owner:** Rodrigo · **Complexity:** Moderate (3 components, 0 new dependencies, custom ranking
and threshold logic) · **Branch:** `story/s4.5/priority-formula`
**Hands off to:** s4.5-API (Nano)

## Problem

A collections team has more accounts than hours. Scores say who is likely to pay; they do not say
who to call first — a 95-score account owing $200 is worth less attention than a 30-score account
owing $100,000.

## Value

The portfolio ordered by expected recoverable pesos, and the subset holding most of that value
identified, so a limited team works the accounts that matter.

## Measurement (taken before designing)

Expected recoverable value = `outstanding × score/100`, computed over real scored portfolios,
400 clients, seed 42.

### The portfolio is **not** Pareto-distributed

| Top X% of clients | Share of expected value |
|---:|---:|
| 5% | 13.2% |
| 10% | 23.7% |
| **20%** | **40.1%** |
| 30% | 53.5% |
| 50% | 73.4% |
| **58%** | **80%** |
| 80% | 93.1% |

Consistent across all three sectors (top-20% share: manufacturing 39.9%, retail 40.1%,
professional services 40.5%).

**The classic "20% of accounts hold 80% of value" is false for this data.** Reaching 80% of
expected value requires ~58% of the portfolio.

**Cause is upstream, not in this formula.** `ProceduralGenerator` draws invoice amounts from
`Normal(amount_mean, amount_std)` — with the demo's 10,000 ± 3,000 that is a tight distribution
with no heavy tail, so every client holds a similar balance and concentration can only come from
score variation. Real receivables portfolios are heavy-tailed: a few large clients dominate, which
is exactly what makes Pareto filtering valuable.

**Decision: implement the filter honestly and raise the generator finding separately** (D3).

## Decisions

### D1 — Priority = expected recoverable value, in pesos

`expected_recoverable = outstanding_amount × score / 100`

This has real units **only because s4.3 D1 kept `predict_proba` calibrated** (no `class_weight`).
`score/100` is a genuine probability of collection within 90 days, so the product is money. Had
the model been balanced for accuracy, this would be an index and the Pareto claim would be
arithmetic without meaning.

`outstanding_amount` already nets partial payments (s4.2 gemba finding), so exposure is the true
balance rather than the invoiced total.

### D2 — Pareto returns the smallest prefix reaching the threshold

Sort by expected value descending, accumulate, cut at the first client whose cumulative share
reaches 80%. "Smallest prefix" is the operative requirement: any larger set also exceeds 80% and
would be trivially correct and useless.

The threshold is a **parameter defaulting to 0.80**, not a constant — the measured curve has no
knee, so 80% is a business choice rather than a property of the data.

### D3 — Report actual concentration, never assume it

The result carries `client_count`, `portfolio_count` and `value_share`. An operator sees "158 of
271 accounts hold 80.2% of recoverable value" — the truth for this portfolio — rather than an
inherited "20/80" claim the data does not support.

**Raised separately for the team:** if the demo needs a compelling concentration story, the fix is
a log-normal amount distribution in E3's generator, not a tuned threshold here. Parked, not
silently applied — changing it would invalidate seed-42 fixtures that s4.2, s4.3 and s4.4 tests
pin.

### D4 — `ScoringRun` carries outstanding per client

`Score` has no `outstanding_amount`, so `ScoringRun` alone cannot compute priority. s4.3's use
case already holds the feature frame, so exposing it costs nothing there and makes the object
self-contained.

**This is the contract Nano builds s4.5-API against**: one input object, one definition of
"outstanding". The alternative — passing `RawDataset` alongside and recomputing — would put a
second exposure calculation in the system, and s4.2 already showed what two definitions of the
same derived quantity cost.

Additive change to a frozen dataclass; no behaviour change to scoring.

## Gemba findings

| Finding | Consequence |
|---|---|
| `ScoringRun` holds `scores`, `evaluation`, `unscored_client_count` — no exposure | D4 |
| `Score` is frozen with no amount field | Not modified; priority is derived, not stored on the entity |
| `outstanding_by_client()` already exists and nets partial payments | Reused via `ScoreScenario`, not reimplemented |
| No prioritisation, Pareto or ranking code anywhere | Net-new |
| Clients without outstanding balance are already absent from `scores` (ADR-006 D4) | They cannot appear in the queue; no filtering needed here |

### Legacy sweep

**Nothing becomes orphaned — net-new.** No prior prioritisation exists. `ScoringRun` gains a
field; it is not superseded, and no second version is introduced.

### Drift risk

`governance/drift-hotspots.json` absent. **AG2 (clone amplification):** exposure must have one
definition. D4 exists precisely to avoid a second one — the failure mode s4.2 hit when the
labeller and extractor both needed "what is still owed".

## Approach

| Component | Change | Purpose |
|---|---|---|
| `app/domain/value_objects/prioritized_case.py` | create | `PrioritizedCase`, `PrioritizedPortfolio` |
| `app/application/services/prioritizer.py` | create | Ranking + Pareto filter |
| `app/application/use_cases/prioritize_scenario.py` | create | Consumes a `ScoringRun` |
| `app/application/use_cases/score_scenario.py` | modify | `ScoringRun` gains `outstanding_by_client` (D4) |

**Lean gates.** KISS: sort, cumulative sum, cut — no optimisation library. DRY: exposure comes
from the existing helper via `ScoreScenario`. YAGNI: no workload balancing, no time-discounting,
no collector capacity model. MVP: an ordered list plus a subset.

**Supply chain:** N/A — no new dependency.

**Missing-data strategy:** a client in `scores` but absent from `outstanding_by_client` would be a
bug in s4.3, not a condition to tolerate — it raises rather than defaulting to zero, because a
zero would silently rank a real account last.

## Examples

```python
from app.application.use_cases.prioritize_scenario import PrioritizeScenario

portfolio = PrioritizeScenario().execute(scoring_run, threshold=0.80)

portfolio.cases[:3]
# [PrioritizedCase(client_id=..., score=30.4, outstanding=101_200.0,
#                  expected_recoverable=30_764.8, rank=1),
#  PrioritizedCase(client_id=..., score=88.1, outstanding=33_050.0,

```


## Acceptance criteria

Refines the Gherkin in `s4.5-formula-story.md`.

**MUST**

1. `expected_recoverable == outstanding × score / 100`, in pesos
2. A 30-score/$100k account outranks an 88-score/$33k account
3. A 95-score/$200 account does not appear above materially larger expected values
4. Pareto returns the **smallest** prefix whose cumulative share reaches the threshold
5. The subset's share is ≥ threshold, and reported as an actual figure
6. Single-dominant-client portfolio returns that client alone
7. Same seed → identical values **and** ranks; ties break deterministically
8. A client in `scores` missing from `outstanding_by_client` raises, never defaults to 0

**SHOULD**

9. `summary()` states the real concentration in Spanish
10. Threshold is a parameter, defaulting to 0.80

**MUST NOT**

11. No tuning of the threshold to manufacture a flattering subset size
12. No second definition of "outstanding" — it comes from `ScoringRun`
13. No unused public API or parameter — **checked before close**, after three consecutive stories
    introduced one

## Open questions

- **Expected-value ranking buries small accounts permanently.** A $200 account with a 95 score may
  never surface, however cheap it is to collect. That is a workload-policy question for E5, not a
  formula change — noted so it is a decision rather than an accident.
- The demo's Pareto story depends on a generator change (D3) that this story deliberately does not
  make.
