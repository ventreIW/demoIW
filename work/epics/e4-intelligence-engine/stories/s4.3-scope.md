# s4.3 Scope ŌĆö Collectability Scoring Model

**Owner:** Rodrigo ┬Ę **Size:** M ┬Ę **Branch:** `story/s4.3/collectability-model`

## In Scope

- `scikit-learn` added as a backend dependency (ADR-007)
- `IScoringPort` ŌĆö `fit`, `predict_scores`, `explain` contract
- `SklearnScorer` adapter implementing it with `LogisticRegression`
- 0ŌĆō100 score derived from `predict_proba`
- High / Medium / Low categorization, thresholds justified against the observed
  score distribution
- Evaluation on held-out clients: **ROC-AUC and balanced accuracy**, with the
  majority-class baseline reported alongside
- Quality gate: a model that fails to beat the baseline raises rather than ships
- `ScoreScenario` use case: build training set ŌåÆ fit ŌåÆ score ŌåÆ return `Score` entities
- Coefficient direction sanity assertions (`days_overdue_max` negative,
  `pct_invoices_settled` positive)
- **Drop `sector` from the design matrix** ŌĆö constant within a scenario, zero variance
- **Raise `GenerationParams.client_count` minimum** so unscoreable scenarios cannot be created
- Clear, actionable error when a scenario is too small to score

## Risks

| Risk | Mitigation |
|---|---|
| **Model fails to beat the baseline.** M1 is a go/no-go; s4.4/s4.5/s4.6 all rest on this | Stop and revise ADR-007 rather than proceeding. Do not tune until the simple model is understood |
| Raising the `client_count` minimum breaks existing E3 tests or fixtures | Run the full backend suite; E3 tests use 3ŌĆō400 clients, so most are unaffected. Any using fewer must be updated deliberately, not silently |
| Category thresholds chosen to flatter the demo | Justify against the observed distribution and record it, per the epic acceptance gate |
| `scikit-learn` pulls `scipy` (~60 MB) | Accepted in ADR-007; noted for deployment |
| Persistence gap while s4.9 is unmerged | Use case returns `Score` entities; the repository call is the only thing deferred |
