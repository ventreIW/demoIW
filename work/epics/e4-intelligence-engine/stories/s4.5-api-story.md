# Story: Prioritized List API Endpoint

**Owner:** Nano · **Branch:** `story/s4.5/prioritized-list-endpoint`
**Prerequisite:** s4.5-formula contract agreed (T1 in 5d8980f)

## User Story

As a **collections manager**,
I want a sortable, filterable API endpoint that returns the prioritized portfolio,
so that the operations team can consume it directly in their queue UI (E5).

## Acceptance Criteria

### Scenario: Prioritized list endpoint returns ranked portfolio with Pareto subset

```gherkin
Given an active scenario with scored clients
When GET /api/v1/scenarios/{scenario_id}/prioritized is called
Then the response contains the full ranked portfolio
And the Pareto subset holding ≥80% of expected recoverable value
And each case includes client_id, score, outstanding, rank, expected_recoverable
And the response reports the actual concentration (not an assumed 20/80)
```

### Scenario: Endpoint supports sorting

```gherkin
Given a prioritized portfolio
When the endpoint is called with ?sort=outstanding&order=desc
Then the full portfolio is returned sorted by outstanding amount descending
And the Pareto subset remains the smallest prefix reaching the threshold
```

### Scenario: Endpoint supports filtering by category

```gherkin
Given a prioritized portfolio
When the endpoint is called with ?category=High
Then only High-category cases are returned (with their correct ranks)
And the Pareto subset is computed on the filtered set
```

### Scenario: Endpoint supports filtering by days overdue

```gherkin
Given a prioritized portfolio
When the endpoint is called with ?days_overdue_min=30
Then only cases with days_overdue ≥ 30 are returned
```

### Scenario: Empty or unscored scenario returns graceful response

```gherkin
Given a scenario with no scores yet
When the endpoint is called
Then it returns 404 with a clear message
```

### Scenario: Contract shape matches s4.5-formula handoff

```gherkin
Given the s4.5-formula contract (PrioritizedCase, PrioritizedPortfolio)
When the endpoint serializes the response
Then every field matches the documented contract exactly
And expected_recoverable is in pesos (outstanding × score/100)
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| Active scenario with scores | GET /api/v1/scenarios/{id}/prioritized | 200 with PrioritizedPortfolio |
| ?sort=amount&order=desc | Sort by outstanding | Portfolio sorted by amount desc |
| ?category=High | Filter by category | Only High cases, ranks preserved |
| ?days_overdue_min=60 | Filter by overdue | Only cases ≥60 days overdue |
| ?threshold=0.90 | Override Pareto threshold | Subset reaches ≥90% |

## Notes

- **This story is the API only.** The formula (expected recoverable, ranking, Pareto) is defined in s4.5-formula (Rodrigo) and handed off via `PrioritizedCase` / `PrioritizedPortfolio` value objects in commit 5d8980f.
- The use case `PrioritizeScenario` already exists and consumes a `ScoringRun`. This story wires it to an HTTP endpoint.
- The Pareto threshold is configurable via query param (default 0.80 = 80%).
- No persistence — priorities are derived on read from scores + outstanding.
- Contract fields: `client_id`, `score`, `outstanding`, `rank`, `expected_recoverable` (derived property), `cases[]`, `pareto_subset[]`, `threshold`, `total_expected_recoverable`, `subset_expected_recoverable`, `portfolio_count`, `subset_count`, `value_share`, `summary()`.
- Frontend (E5) will consume this exact shape; treat changes as contract changes.
