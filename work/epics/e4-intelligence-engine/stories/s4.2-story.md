# Story: Feature Engineering & Training Set

**Owner:** Rodrigo
**Branch:** `story/s4.2/feature-engineering`

## User Story

As the **collectability scoring engine**,
I want a labelled, leakage-free training set built from a generated scenario,
so that a model can learn to predict whether a client's outstanding balance will be
collected — rather than learning to invert the data generator.

## Acceptance Criteria

### Scenario: Feature extraction from a generated scenario

```gherkin
Given a RawDataset produced by ProceduralGenerator with 100 clients
When the feature extractor runs
Then it returns one feature row per client
And every feature is derived only from observable invoices, payments and sector
And the frame contains no "payment_history_pattern" column
```

### Scenario: Leakage guard is enforced, not assumed

```gherkin
Given a RawDataset whose clients carry a payment_history_pattern column
When the feature extractor runs
Then "payment_history_pattern" is absent from the returned columns
And the test fails loudly if it is ever reintroduced
```

### Scenario: Forward outcome labelling

```gherkin
Given a client whose hidden pattern is "on_time"
And a client whose hidden pattern is "default"
When the 90-day outcome labeller runs over many such clients
Then the on_time cohort has a materially higher collection rate than the default cohort
```

### Scenario: Reproducibility for a fixed seed

```gherkin
Given a scenario generated with seed 42
When the training set is built twice from that scenario
Then both runs produce identical features and identical labels
```

### Scenario: Split is by client, not by row

```gherkin
Given a labelled training set of 100 clients
When it is split into train and test partitions
Then no client id appears in both partitions
And both partitions contain at least one example of each label class
```

### Scenario: Degenerate scenario is rejected clearly

```gherkin
Given a scenario so small or uniform that only one label class is present
When the training set is built
Then a clear domain error is raised naming the problem
And it does not silently return an unusable single-class training set
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|---|---|---|
| Client with 4 invoices, 3 settled, max 12 days overdue | extract features | `pct_invoices_settled=0.75`, `days_overdue_max=12.0`, `invoice_count=4` |
| Client with 0 open invoices | extract features | `outstanding_amount=0.0`, `days_overdue_max=0.0` — no NaN |
| Client with a partial payment on an open invoice | extract features | `has_partial_payments=True` |
| Clients DataFrame containing `payment_history_pattern` | extract features | column **absent** from output |
| 1000 `on_time` clients vs 1000 `default` clients | label at 90-day horizon | on_time collection rate materially exceeds default |
| Same scenario, seed 42, built twice | build training set | byte-identical features and labels |
| 100 clients | split by client | train ∪ test = 100, train ∩ test = ∅ |

## Notes

- Implements **ADR-006** (prediction target + leakage guard). The ADR is written; this story
  carries it into code.
- The labeller **imports `_PATTERN_PROFILES` from `ProceduralGenerator`** rather than redefining
  the constants. One source of truth for the causal model (ADR-004).
- Labelling uses the scenario's seeded RNG so training sets are reproducible.
- Feature list is specified in `work/epics/e4-intelligence-engine/design.md` § Key contracts.
- No model training here — that is s4.3. This story ends with a `TrainingSet` in hand.
- The leakage test is the single most important artifact in this story. Without it the
  guard is a comment, not a constraint.
