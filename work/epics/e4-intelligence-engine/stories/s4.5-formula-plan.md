# s4.5-formula Plan — Priority Value & Pareto Threshold

**Story:** s4.5-formula · **Size:** S · **Owner:** Rodrigo · **Date:** 2026-07-21
**Design:** `s4.5-formula-design.md` · **Branch:** `story/s4.5/priority-formula`

3 tasks, ordered to **unblock Nano first** rather than risk-first.

## Ordering rationale — deliberate deviation

Risk-first would put the Pareto filter (T2) first: it holds the only real logic risk, since
"smallest qualifying prefix" is where an off-by-one lives.

**T1 goes first anyway**, because s4.5-API is blocked on the *shape*, not the logic. Nano is
currently building against a design document; T1 gives him importable types within the hour. The
Pareto risk is contained — it is one function with a measured expectation to check against — so
paying a small ordering cost to unblock a second developer is the better trade.

T1 is pushed **as soon as it is green**, before T2 begins.

## Risks

| Risk | Mitigation |
|---|---|
| **Off-by-one in "smallest prefix"** — the only real logic risk | `test_pareto_returns_smallest_qualifying_prefix` proves minimality by removal, not by threshold alone |
| Contract changes after Nano starts building | T1 is pushed before T2 begins, so the shape is fixed early. Any later change must be told to him, not merged quietly |
| The queue is arithmetically right and reads wrong to a collections manager | T3 prints the top 20 for human reading — s4.4's lesson |
| An unused public API slips in again | AC #13; checked **before** close, after three consecutive stories introduced one |
| `ScoringRun` change conflicts with Nano's work | The field is mine to add; he consumes it. Flagged in the handoff |
