## What to improve

- **Path resolution in test files:** Using `__dirname` with `../../../../frontend/public/` is fragile. A utility using `process.cwd()` or a vitest root alias would be more robust. The initial commit had a wrong depth (4 vs 5 levels up).
- **WSL integration testing:** The dev server (`npm run dev`) and Lighthouse audit are blocked by WSL cross-filesystem slowness on `/mnt/c/`. Structural verification was the pragmatic fallback, but a real browser test would be ideal.
- **Placeholder icons:** Solid-color PNGs serve the PWA requirements but look unprofessional. Real branded icons should be created before production deployment.
- **Build timeout:** `npm run build` compiled successfully but timed out during the lint/types post-phase on WSL. A CI pipeline would handle this, but local development suffers.

## Heutagogical Checkpoint

1. **What did I learn?**
   Testing PWA static assets with vitest and `fs.readFileSync` is a practical alternative to browser-based Lighthouse audits for CI. Next.js 15 Metadata API accepts `manifest` and `themeColor` directly (no separate Viewport export needed for simple cases). WSL `/mnt/c/` makes the Next.js dev server and linting unusably slow — structural verification must substitute.

2. **What would I change about the process?**
   Use `path.resolve(process.cwd(), 'public/manifest.json')` instead of `__dirname` navigation in vitest tests. The `process.cwd()` equals the vitest root (the frontend directory) and eliminates fragile `../` chains.

3. **Framework improvements?**
   The RaiSE gate system (`rai gate check gate-tests`) is designed for pytest/Python and doesn't work for Next.js/vitest frontends in a monorepo. A frontend gate adapter that runs `npx vitest run --scope <path>` would close this gap. The `--scope` flag on `rai gate check` also doesn't prevent conftest discovery in sibling packages.

4. **What am I more capable of now?**
   Setting up a complete PWA stack for Next.js 15 from scratch (manifest, service worker, metadata, icons). Writing vitest tests that validate static JSON/JS files via filesystem reads. Navigating the Next.js Metadata type system for PWA configuration fields.
