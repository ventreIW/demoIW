**Story ID:** s3.4  
**Size:** S

## Overview

This plan describes the backend implementation for story s3.4: Generate Scenario Endpoint & UI. The backend work includes defining repository interfaces, implementing SQLAlchemy repositories for clients, invoices, and payments, updating the GenerateDataset use case to persist generated data, adding the POST /api/scenarios/generate endpoint, wiring dependencies in the container, and writing an integration test.

## Execution Order & Rationale

1. **T1** must come first because all other tasks depend on the repository interfaces.
2. **T2, T3, T4** can be done in any order after T1; they are independent of each other.
3. **T5** depends on T1-T4 (interfaces and uses the repository interfaces.
4. **T6** depends on T1-T4 (containers provides the repository implementations).
5. **T7** depends on T5 (use case updated) and T6 (container wired) because the endpoint needs the use case instance.
6. **T8** depends on T7 (endpoint exists) and can be written after the endpoint is ready.
7. **T9** is the final validation step.

Rationale: This order minimizes blocking dependencies and allows parallel work on the three repository implementations (T2-T4) after T1.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Missing dependencies in repository interfaces (e.g., needed methods) | Write tests first (TDD) to discover required methods; iterate quickly. |
| Incorrect mapping from DataFrame to domain entities | Reuse existing mappers from the project (e.g., in `backend/app/adapters/persistence/mappers.py`). |
| Circular dependencies in container | Keep container registration simple; use interfaces (not concrete classes) for dependencies. |
| Performance issues with bulk inserts | Use SQLAlchemy's `insert()` bulk insertion as noted in the design. |
| Integration test flakiness due to database state | Use transaction rollback or fixture-based database cleanup in tests. |

## Gate Policy (Mandatory)

| When | Command | Scope |
|------|---------|-------|
| Per task | `rai gate check gate-tests --scope <test_dir>/` | Only tests for the changed module (e.g. `backend/tests/unit/` for unit tests, `backend/tests/integration/` for integration tests) |
| Per task | `rai gate check gate-lint` | Full project (fast) |
| Per task | `rai gate check gate-format` | Full project (fast) |
| Per task | `rai gate check gate-types` | Full project |
| End of story | `rai gate check gate-tests --scope backend/tests/` | Changed package only — NOT full suite |
| MR creation | `/rai-mr-create` | Full suite — the ONLY place where unscoped tests run |

**NEVER run test commands directly** (e.g. `uv run pytest`, `npm test`) — always use `rai gate check`. Direct invocation floods context with raw output and bypasses gate tracking.

Each task's Verification section MUST use `rai gate check gate-tests --scope <path>` with the concrete test path for that task. Generic or unscoped test commands are not acceptable.
