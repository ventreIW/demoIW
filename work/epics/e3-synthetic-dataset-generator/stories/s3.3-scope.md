# s3.3 Scope — LLM Enrichment Layer

## In Scope
- `prompts/data_enrichment/v1_company_description.txt` — prompt template
- `app/application/services/llm_enrichment_service.py` — `LLMEnrichmentService`
- `app/application/use_cases/generate_dataset.py` — `GenerateDataset` use case (wires procedural + enrichment)
- `tests/unit/test_llm_enrichment_service.py` — mocked ILLMPort tests
- ADR-004 (LLM batching strategy documentation)

## Out of Scope
- Persisting the enriched data to the database (s3.4)
- The API endpoint (s3.4)
- Communications prompt templates (E5 — separate prompt directory)
- Enrichment of invoices or payments (only clients are enriched)
- Real-time progress reporting during enrichment
- Caching of LLM responses between enrichment runs

## Done when
- LLMEnrichmentService successfully enriches client data with sector-appropriate names and descriptions
- Enrichment processes data in configurable batches (default 20) to optimize LLM API usage
- Service gracefully degrades to original Faker names when LLM calls fail
- Prompt template is loaded from external file rather than being hardcoded
- Generated company names when LLM calls
- Prompt template is loaded from external file rather than being hardcoded
- Generated company names are demonstrably different from original Faker names (enrichment verified)
- All unit tests use mocked ILLMPort with zero actual OpenRouter API calls
- GenerateDataset use case properly wires together ProceduralGenerator and LLMEnrichmentService
- ADR-004 is written and approved documenting the batching strategy and rationale
