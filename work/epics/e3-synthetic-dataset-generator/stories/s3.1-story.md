# Story: Procedural Data Generation Layer

## User Story
As a developer,
I want a seed-based procedural generator that produces statistically realistic client, invoice, and payment data,
so that every generated scenario is reproducible and distribution-appropriate for its sector.

## Acceptance Criteria

### Scenario: Generator returns a complete RawDataset
```gherkin
Given a valid GenerationParams with a fixed seed
When ProceduralGenerator(params).generate() is called
Then it returns a RawDataset containing clients, invoices, and payments DataFrames
And the clients DataFrame has exactly params.client_count rows
```

### Scenario: Generation is deterministic
```gherkin
Given the same GenerationParams (identical seed and parameters)
When generate() is called twice
Then both calls return identical clients, invoices, and payments DataFrames
```

### Scenario: Invoice amounts follow a Normal distribution
```gherkin
Given GenerationParams with amount_mean and amount_std
When invoices are generated
Then every invoice amount is positive (no negative or zero amounts)
And the sample mean is within 15% tolerance of amount_mean
```

### Scenario: Payment pattern causally drives overdue behaviour
```gherkin
Given a large portfolio grouped by each client's payment_history_pattern
When invoice overdue rates are compared across patterns
Then overdue propensity rises monotonically from ON_TIME to DEFAULT
And ON_TIME clients are overdue < 15% while DEFAULT clients are overdue > 75%
And average days_overdue tracks each pattern's severity
```

### Scenario: Invoice records are internally coherent
```gherkin
Given a generated set of invoices
When an invoice status is inspected
Then overdue invoices have days_overdue >= 1 and paid invoices have days_overdue = 0
And issue_date always precedes due_date
```

### Scenario: Payments reflect settlement behaviour
```gherkin
Given a generated dataset
When payments are inspected
Then every settled ("paid") invoice has exactly one full-amount payment
And overdue invoices of PARTIAL payers carry a payment below the invoice amount
And overdue invoices of DEFAULT clients carry no payment
And no payment is dated after the reference date
```

### Scenario: Payment pattern is sector-weighted
```gherkin
Given two datasets generated for MANUFACTURING and RETAIL sectors
When client payment_history_pattern distributions are compared
Then MANUFACTURING has a higher DELAYED_30 (slow-payer proxy) proportion than RETAIL
```

## Examples (Specification by Example)

| Input (params) | Action | Expected Output |
|----------------|--------|-----------------|
| seed=42, client_count=50 | generate() → clients | clients DataFrame has exactly 50 rows |
| seed=42 (run A) vs seed=42 (run B) | generate() twice | DataFrames compare equal |
| amount_mean=10000, amount_std=3000 | generate invoices | all amounts > 0; sample mean ∈ [8500, 11500] |
| pattern=ON_TIME vs DEFAULT, N large | compare overdue rate | ON_TIME < 0.15; DEFAULT > 0.75 |
| pattern=PARTIAL, overdue invoice | inspect payment | payment amount < invoice amount (~20–60%) |
| sector=MANUFACTURING vs RETAIL | compare DELAYED_30 share | MANUFACTURING (0.30) > RETAIL (0.20) |

## Notes
- Overdue behaviour is **not** a global knob — each client's `payment_history_pattern` drives its
  invoices' overdue propensity, ageing, and payments (see ADR-004). Overall overdue rate emerges
  from the sector's pattern mix.
- Config (sector weights, per-pattern `_PatternProfile`, amount distribution) is hardcoded in the
  generator, not DB-driven — see `s3.1-design.md`.
- Determinism anchors: `np.random.default_rng(seed)` + `Faker.seed(seed)`; UUIDs derive from the
  seeded RNG; `reference_date` pins invoice ageing for cross-day reproducibility.
- Faker company names are placeholders here; s3.3 replaces them with LLM-enriched names.
- Non-functional: `mypy` strict passes on all new files. ADR-003 = procedural + LLM hybrid;
  ADR-004 = pattern-driven behavioural model.
- Refs: epic `design.md`, `s3.1-design.md`, `s3.1-scope.md`.
