# s3.1 Design — Procedural Data Generation Layer

**Story:** s3.1 · **Epic:** E3 — Synthetic Dataset Generator
**Complexity:** Moderate (3–4 components, custom statistical logic, 0 external integrations)
**Branch:** `story/s3.1/procedural-generator`

---

## Problem

E3 needs a reproducible source of realistic accounts-receivable data before any LLM
enrichment (s3.3) or persistence (s3.4) can happen. Without a deterministic procedural
layer, generated scenarios would be non-reproducible and demos non-repeatable.

## Value

A seed-based generator makes every scenario **reproducible** (same seed → same data) and
**distribution-appropriate** per sector, so facilitators can regenerate identical demo
datasets and downstream scoring (E4) sees statistically realistic inputs.

## Approach

A single synchronous `ProceduralGenerator` implementing `IDatasetPort`, seeded by
`np.random.default_rng(seed)` for numeric draws and `Faker.seed(seed)` for names, producing
three pandas DataFrames wrapped in a frozen `RawDataset` value object. Sector-specific
payment-pattern weights are **hardcoded constants** in the generator (not DB-driven — YAGNI
for a demo asset).

**Lean gates:** KISS — one class, three private builders. DRY — no existing generator to
reuse. YAGNI — no config table, no streaming, no persistence. MVP — DataFrames only; entity
mapping and DB writes are s3.4.

**Supply chain gate:** `numpy`, `pandas`, `faker` are all widely-adopted, well-maintained
packages (N/A for deep vetting per skill skip rule). They are core to the declared project
stack (CLAUDE.md). Isolation: confined to `app/adapters/dataset/` — numeric/df types do not
leak past the `RawDataset` boundary into domain entities.

### Sector payment-pattern weights (remapped to real enum)

Probabilities over the real `PaymentPattern` values; each row sums to 1.0:

| Sector | ON_TIME | DELAYED_30 | DELAYED_60 | DELAYED_90_PLUS | PARTIAL | DEFAULT |
|---|---|---|---|---|---|---|
| MANUFACTURING | 0.30 | 0.30 | 0.15 | 0.08 | 0.10 | 0.07 |
| RETAIL | 0.25 | 0.20 | 0.15 | 0.10 | 0.20 | 0.10 |
| PROFESSIONAL_SERVICES | 0.45 | 0.20 | 0.10 | 0.05 | 0.15 | 0.05 |

Sector-weighting AC uses **DELAYED_30 as the slow-payer proxy**: MANUFACTURING (0.30) >
RETAIL (0.20), a stable, testable inequality.

### Per-pattern behavioural profiles (causal — see ADR-004)

Each pattern drives its clients' invoice outcomes. Overdue propensity is monotonic so E4 has
learnable signal; a portfolio's overall overdue rate emerges from the sector's pattern mix.

| Pattern | overdue_prob | late_days_mean | overdue payment behaviour |
|---|---|---|---|
| ON_TIME | 0.05 | 3 | — |
| DELAYED_30 | 0.25 | 30 | none yet |
| DELAYED_60 | 0.45 | 60 | none yet |
| DELAYED_90_PLUS | 0.65 | 100 | none yet |
| PARTIAL | 0.60 | 50 | partial payment (20–60%) |
| DEFAULT | 0.90 | 160 | none |

Profile numbers are hand-authored for clear inter-pattern separation in demos — plausible, not
fitted to real data (the first thing to revisit for a client wanting sector-accurate figures).

## Design decisions

| ID | Decision | Rationale |
|---|---|---|
| D1 | `IDatasetPort.generate()` is **synchronous** | Generation is CPU-bound (numpy/pandas); no I/O. Async would add ceremony with no benefit and diverge from the actual workload. |
| D2 | Distribution config **hardcoded** in the generator | Demo asset; DB-driven config is speculative (YAGNI). Constants keep determinism auditable. |
| D3 | `RawDataset` holds **pandas DataFrames**, frozen VO | Vectorised generation + stats live in df-space; freezing the VO prevents accidental mutation between generate() and downstream consumers. |
| D4 | Raw invoice `status ∈ {"overdue","paid"}` as strings | No `InvoiceStatus` enum exists; raw layer stays string-typed, canonical mapping deferred to s3.4. |
| D5 | Dual determinism anchor: `default_rng(seed)` + `Faker.seed(seed)` | Faker draws from its own RNG; seeding both is required for full reproducibility (numeric + names). |
| D6 | **Payment pattern is causal** (per-pattern `_PatternProfile`) — no global `overdue_rate` | The pattern must drive overdue propensity, ageing, and payments, else there is no signal for E4 to learn and no realism. Overall overdue rate emerges from the sector mix. See ADR-004. |
| D7 | UUIDs derived from the seeded RNG (`rng.bytes(16)`), not `uuid.uuid4()` | `uuid4()` reads OS entropy and breaks determinism even when all numeric draws are seeded. |
| D8 | `reference_date` param anchors invoice ageing | Makes reproducibility a property of the params, not the wall clock; `None` → `date.today()` for a live-feeling demo. |

## Examples (Specification by Example)

### Invocation

```python
from app.application.use_cases.generate_dataset import GenerationParams
from app.adapters.dataset.procedural_generator import ProceduralGenerator
from app.domain.enums import Sector

params = GenerationParams(
    seed=42,
    sector=Sector.MANUFACTURING,
    client_count=50,
    invoice_volume=4,        # Poisson mean invoices/client
    amount_mean=10_000.0,
    amount_std=3_000.0,
    reference_date=date(2026, 7, 2),   # optional; None -> date.today()
)
ds = ProceduralGenerator(params).generate()   # -> RawDataset
```

### Expected output (shape)

```python
ds.clients.shape[0] == 50                       # exactly client_count
set(ds.clients.columns) >= {"id", "name", "sector", "payment_history_pattern"}
set(ds.invoices.columns) >= {"id", "client_id", "folio", "amount",
                             "issue_date", "due_date", "days_overdue", "status"}
(ds.invoices["amount"] > 0).all()               # no non-positive amounts
set(ds.invoices["status"].unique()) <= {"overdue", "paid"}
# every SETTLED invoice has a full-amount payment; overdue rate rises with pattern severity
paid_ids = set(ds.invoices.loc[ds.invoices.status == "paid", "id"])
assert paid_ids <= set(ds.payments["invoice_id"])
```

### Determinism

```python
a = ProceduralGenerator(params).generate()
b = ProceduralGenerator(params).generate()
pd.testing.assert_frame_equal(a.clients, b.clients)
pd.testing.assert_frame_equal(a.invoices, b.invoices)
pd.testing.assert_frame_equal(a.payments, b.payments)
```

### Data structures

```python
class GenerationParams(BaseModel):
    seed: int
    sector: Sector
    client_count: int = Field(gt=0)
    invoice_volume: float = Field(gt=0)         # Poisson mean
    amount_mean: float = Field(gt=0)
    amount_std: float = Field(ge=0)
    reference_date: date | None = None          # ageing anchor; None -> today()
    model_config = {"frozen": True}

@dataclass(frozen=True)
class RawDataset:
    clients: pd.DataFrame
    invoices: pd.DataFrame   # id, client_id, folio, amount, issue_date, due_date, days_overdue, status
    payments: pd.DataFrame   # id, invoice_id, amount, paid_date
```

## Acceptance Criteria

Gherkin scenarios live in `s3.1-story.md`. Severity mapping:

- **MUST** — determinism; `client_count` exactness; positive amounts; **pattern drives overdue**
  (ON_TIME < 0.15, DEFAULT > 0.75, monotonic); record coherence (overdue⇒days≥1, paid⇒days=0,
  issue<due); settled invoice ⇒ full payment; PARTIAL overdue ⇒ partial payment; DEFAULT overdue
  ⇒ no payment; sector weighting (MANUFACTURING DELAYED_30 > RETAIL); `mypy` strict clean;
  ADR-003 + ADR-004 written.
- **SHOULD** — amount sample mean within 15% of `amount_mean`; no payment after `reference_date`.
- **MUST NOT** — no negative/zero amounts; no DB writes; no LLM calls; no API/frontend code.

## Testing strategy

| Layer | Test | Purpose |
|---|---|---|
| unit | `test_determinism` | same seed+params → `assert_frame_equal` on all three DataFrames |
| unit | `test_client_count_matches_params` | rows == `params.client_count` |
| unit | `test_amounts_positive` | `(amount > 0).all()`, min ≥ 100 |
| unit | `test_amount_mean_within_tolerance` | sample mean within 15% |
| unit | `test_pattern_drives_overdue_propensity` | ON_TIME < DELAYED_60 < DEFAULT; ON_TIME<0.15, DEFAULT>0.75 |
| unit | `test_days_overdue_coherent_with_status` | overdue⇒days≥1, paid⇒days=0 |
| unit | `test_settled_invoice_has_full_payment` | one full-amount payment per paid invoice |
| unit | `test_partial_payers_make_partial_payments_on_overdue` | PARTIAL overdue payment < amount |
| unit | `test_default_overdue_invoices_have_no_payment` | DEFAULT overdue ⇒ no payment row |
| unit | `test_dates_are_coherent` | issue<due; paid_date ≤ reference_date |
| unit | `test_sector_weighting_*` | MANUFACTURING DELAYED_30 share > RETAIL (large N) |

## Resolved during implementation

- Invoice columns now fully align with the `Invoice` entity (`folio`, `issue_date`,
  `days_overdue` included) → near-zero mapping friction for s3.4.
- Payments columns fixed at `{id, invoice_id, amount, paid_date}`.
- Overdue behaviour moved from global knobs to per-pattern profiles (ADR-004) after review
  flagged the pattern label was decorative.
