# s3.1 Scope — Procedural Data Generation Layer

## In Scope

- `app/application/use_cases/generate_dataset.py` — `GenerationParams` model
- `app/adapters/dataset/procedural_generator.py` — `ProceduralGenerator` (implements `IDatasetPort`)
- `app/ports/dataset_port.py` — `IDatasetPort`, `RawDataset` types
- `app/domain/value_objects/raw_dataset.py` — `RawDataset(clients, invoices, payments)`
- Sector-specific distribution config (hardcoded constants in generator, not DB-driven)
- `tests/unit/test_procedural_generator.py` — all acceptance criteria tests
- ADR-003 in `dev/decisions/`

## Out of Scope

- LLM enrichment (s3.3)
- OpenRouter adapter (s3.2)
- Any API endpoint
- Persistence to database (s3.4)
- Frontend

## Done when

- `ProceduralGenerator(params).generate()` returns a `RawDataset` with `clients`, `invoices`, `payments` DataFrames.
- Two calls with the same seed + params produce identical DataFrames (determinism test passes).
- Clients DataFrame row count equals `params.client_count` exactly.
- Invoice amounts are all positive and their sample mean is within 15% of `amount_mean`.
- `overdue_rate` fraction of invoices carry `status = OVERDUE` (within 5% tolerance).
- Every non-overdue invoice has a corresponding payment row.
- MANUFACTURING shows a higher `SLOW_PAYER` proportion than RETAIL (sector weighting verified).
- `tests/unit/test_procedural_generator.py` passes (reproducibility, client count, distribution sanity, sector weighting).
- `mypy` passes on all new files.
- ADR-003 written in `dev/decisions/`.
