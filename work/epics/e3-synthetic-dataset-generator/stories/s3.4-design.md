---\nstory_id: \"s3.4\"\nepic_ref: \"E3\"\nsize: \"S\"\ntype: \"code\"\nstatus: \"draft\"\n---\n\n# Story: Generate Scenario Frontend\n\n## User Story\nAs a demo facilitator,\nI want to see a form where I can select a sector and input parameters (client count, seed, etc.),\nclick a \"Generate Scenario\" button,\nand see the generated scenario displayed (or a link to download it),\nso that I can quickly demonstrate the product to prospects.\n\n## Value\nEnables demo facilitators to instantly showcase generated synthetic data to prospects without navigating away from the generator page, improving demo efficiency and clarity.\n\n## Approach\nModify the existing `GenerateScenarioForm` component to:\n1. Call the `generateScenario` API function (to be added) upon form submission.\n2. On success, store the generated scenario summary in state and display:\n   - A success message.\n   - The scenario details (ID, name, sector, client count, status, creation timestamp).\n   - A link to download the scenario data (e.g., as JSON) via a new endpoint `/api/scenarios/{id}/download` (to be implemented by backend story).\n   - The scenario ID appended to the current URL as a query parameter (`?scenarioId=<id>`).\n3. Provide an optional \"Activate & Use\" button that calls `activateScenario` and redirects to `/operations` for users who wish to proceed with the generated scenario.\n4. Maintain loading and error states.\n5. Update the form fields to match the backend's `GenerationParams`:\n   - Keep: sector, client_count, invoice_volume, seed.\n   - Add: amount_mean (default 15000), amount_std (default 8000).\n   - Remove: overdue_rate slider (not part of backend params).\n   - reference_date is optional and set to null (not visible in form).\n\n## Components Affected\n- `frontend/src/components/scenarios/GenerateScenarioForm.tsx` (modify)\n- `frontend/src/lib/api/scenarios.ts` (add `generateScenario` function)\n- `frontend/src/types/scenario.ts` (add `GenerationParams` type)\n\n## Design Decisions\n- **Decision 1: Display scenario summary instead of full data**\n  Rationale: The full dataset may be large; showing a summary (metadata) is sufficient for demo purposes. A download link provides access to the full data if needed.\n- **Decision 2: Use query parameter for scenario ID**\n  Rationale: Allows easy sharing and bookmarking of the generated scenario without requiring state management.\n- **Decision 3: Retain activation option**\n  Rationale: Some users may want to immediately use the scenario; providing an activation button preserves existing workflow.\n- **Decision 4: Align form with backend GenerationParams**\n  Rationale: The frontend must send the correct parameters to the backend endpoint `/api/v1/scenarios/generate`.\n\n## Acceptance Criteria\n### MUST\n- [ ] Form submission calls `/api/v1/scenarios/generate` with correct parameters (sector, client_count, invoice_volume, amount_mean, amount_std, seed, reference_date=null).\n- [ ] On success, displays a message: \"Scenario generated successfully!\"\n- [ ] On success, shows the scenario ID and allows copying it to clipboard.\n- [ ] On success, provides a download link for the scenario data (e.g., JSON format) via `/api/scenarios/{id}/download` (to be implemented by backend story).\n- [ ] On success, updates the URL to include `?scenarioId=<id>`.\n- [ ] Handles loading state (disables form, shows spinner).\n- [ ] Handles error state (shows error message via toast).\n\n### SHOULD\n- [ ] Provides an \"Activate & Use\" button that calls `activateScenario` and redirects to `/operations`.\n- [ ] Optionally displays a preview of the generated data (first 2 rows of clients and invoices) if the API returns it.\n\n### MUST NOT\n- [ ] Redirect to `/operations` immediately after generation without showing the scenario or download link.\n\n## Examples\n### API Usage (TypeScript)\n```typescript\n// src/lib/api/scenarios.ts\nexport async function generateScenario(params: GenerationParams): Promise<ScenarioSummary> {\n  const res = await fetch(`${API_BASE}/api/v1/scenarios/generate`, {\n    method: 'POST',\n    headers: { 'Content-Type': 'application/json' },\n    body: JSON.stringify(params),\n  });\n  if (!res.ok) {\n    throw new Error(`Failed to generate scenario: ${res.status}`);\n  }\n  return res.json(); // Expects { id, name, sector, status, client_count, created_at, ... }\n}\n```\n\n### Success UI\n```\nSuccess! Scenario generated successfully.\n\nScenario Details:\n- ID: scen_123abc\n- Name: Generated Scenario 2026-07-14\n- Sector: Manufacturing\n- Status: inactive\n- Client Count: 100\n\n[Download Scenario as JSON]   [Activate & Use]\n\nURL: http://localhost:3000/en/scenarios/generate?scenarioId=sc_123abc\n```\n\n### Error UI\n```\nError: Failed to generate scenario. Please try again.\n```\n\n## Complexity\nSimple: Involves modifying one component and adding one API function. No complex algorithms or multiple integrations.\n\n## Dependencies\n- Backend endpoint `POST /api/v1/scenarios/generate` must be implemented and return at least a scenario summary.\n- Backend endpoint `GET /api/v1/scenarios/{id}/download` for data download (to be implemented by backend story).\n\n## Legacy Sweep\nThe existing `GenerateScenarioForm` component will be updated in place. No orphaned component.\n\n## Drift Risks\nNone identified: Changes are localized to the scenario generation feature.\n\n## Testing Strategy\n- Unit test the `generateScenario` API function (mock fetch).\n- Integration test the form submission with mocked API calls.\n- Manual verification of UI states (loading, success, error).\n\n## Open Questions\n- What is the exact response format of the `POST /api/v1/scenarios/generate` endpoint? (Assumed to return a `ScenarioSummary`.)\n- Should the download endpoint return JSON or CSV? (JSON is easier to implement and parse.)\n\n---\n## Notes\n- This story depends on the backend story for the `generate` and `download` endpoints.\n- The form fields amount_mean and amount_std are set to default values (15000 and 8000) as per the epic design.\n- The reference_date is set to null, allowing the backend to use the current date.\n
