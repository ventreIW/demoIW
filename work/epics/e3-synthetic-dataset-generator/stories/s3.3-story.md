# Story: S3.3: LLM Enrichment Layer

## User Story
As a demo facilitator,
I want generated companies to have sector-appropriate names and descriptions instead of generic Faker names,
so that the demo feels realistic and convincing to prospects.

## Acceptance Criteria

### Scenario: Successful enrichment of client data
```gherkin
Given a RawDataset containing client records with Faker-generated names
When LLMEnrichmentService.enrich() is called with a valid LLM model
Then each client record has:
  - The original "name" field replaced with an LLM-generated company name
  - A new "sector_description" field added with relevant business description
And the enrichment preserves all other client data fields unchanged
And the total number of client records remains the same
```

### Scenario: Batched LLM API calls
```gherkin
Given a RawDataset containing 47 client records
When LLMEnrichmentService processes the entire dataset with batch size 20
Then the service makes exactly 3 LLM API calls:
  - First call processes clients 1-20
  - Second call processes clients 21-40  
  - Third call processes clients 41-47
And each call includes the appropriate batch of client data in the prompt
And no client is processed in more than one LLM call
```

### Scenario: Graceful degradation on LLM failure
```gherkin
Given a RawDataset with 10 client records
And the LLM service is configured to fail on every 3rd request
When LLMEnrichmentService.enrich() is called
Then:
  - Calls 1, 2, 4, 5, 7, 8, 10 succeed and return LLM-generated names
  - Calls 3, 6, 9 fail and fallback to original Faker names
  - All 10 client records have either LLM-generated or original names
  - No exceptions are propagated to the caller
  - The enrichment process completes successfully for all records
```

### Scenario: External prompt template loading
```gherkin
Given the prompt template file exists at prompts/data_enrichment/v1_company_description.txt
And the file contains: "Generate a company name and description for a {{sector}} business"
When LLMEnrichmentService initializes
Then it loads the template from the file system
And does not contain any hardcoded prompt strings in the source code
And when enrichment runs, it uses the loaded template to construct LLM prompts
```

### Scenario: Verification that enrichment occurred
```gherkin
Given a RawDataset where all client names are known Faker-generated values
When LLMEnrichmentService.enrich() completes successfully
Then:
  - At least 80% of client names have changed from their original values
  - No client name matches the exact original Faker value
  - The new names appear to be valid company names (not random strings)
  - Associated sector descriptions are relevant and coherent
```

### Scenario: Mocked LLM port in unit tests
```gherkin
Given a unit test for LLMEnrichmentService
When the test executes
Then:
  - The ILLMPort dependency is replaced with a mock object
  - All LLM interactions are simulated through the mock
  - Zero actual HTTP requests are made to OpenRouter or any external API
  - Test assertions verify correct behavior based on mock responses
  - Test runs quickly and deterministically without network dependencies
```

### Scenario: Proper use case wiring
```gherkin
Given the GenerateDataset use case
When execute() is called with generation parameters
Then:
  - It first calls ProceduralGenerator.generate() to create RawDataset
  - It then calls LLMEnrichmentService.enrich() on the RawDataset
  - It returns the enriched dataset as the final result
  - No additional processing steps are inserted between generation and enrichment
  - Error handling properly propagates exceptions from either step
```

## Examples (Specification by Example)

| Input Scenario | Batch Size | Expected LLM Calls | Fallback Behavior Verification |
|----------------|------------|-------------------|--------------------------------|
| 5 clients | 20 | 1 call | All use LLM if successful, all fallback if fails |
| 20 clients | 20 | 1 call | All use LLM if successful, all fallback if fails |
| 21 clients | 20 | 2 calls | First 20: LLM/fallback, 21st: LLM/fallback |
| 40 clients | 20 | 2 calls | Two equal batches, each LLM/fallback independently |
| 41 clients | 20 | 3 calls | Two full batches + one partial, each LLM/fallback |
| 0 clients | 20 | 0 calls | No processing, empty result returned |
| 100 clients | 25 | 4 calls | Four equal batches of 25 each |

| LLM Reliability | Expected Outcome | Verification Method |
|-----------------|------------------|---------------------|
| 100% successful | 100% LLM-generated names | Compare output names to original Faker names |
| 0% always fails | 100% original Faker names | Verify fallback to input names occurred |
| 50% random failure | ~50% LLM names, ~50% original names | Statistical distribution check |
| Alternating fail/success | Alternating pattern in output | Pattern matching in result sequence |
| Burst failures (3 consecutive) | 3 original names in a row | Consecutive sequence detection |

| Input Name Pattern | Expected Transformation | Validation Criteria |
|--------------------|-------------------------|---------------------|
| "Acme Corporation" | Different company name | Not equal to input, plausible company name |
| "Globex LLC" | Different company name + description | Name changed, description present and relevant |
| "Initech" | Different company name | Clearly different, not just case change |
| Empty/null name | Generated name + description | Non-empty name, description present |
| Very long name | Different reasonable length name | Output length within typical company name range |

## Notes
- This story implements the LLM enrichment layer as specified in E3 epic scope
- Depends on completion of:
  - s3.1: Procedural data generation layer (provides RawDataset via IDatasetPort)
  - s3.2: OpenRouter adapter (provides ILLMPort LLM interface)
- The enrichment service is designed to be a pure functional transformation:
  - Input: RawDataset + LLM configuration
  - Output: EnrichedDataset (same structure, enhanced name/description fields)
  - No side effects outside of LLM API calls
- Performance considerations:
  - Batch size is configurable to balance API efficiency vs fault tolerance with failure granularity
  - Default batch size of 20 optimizes for typical OpenRouter rate limits
  - Failed batches can be retried independently without re-processing successful batches
- Security considerations:
  - No sensitive data is sent to the LLM (only sector/type information)
  - All prompts are templated and validated before sending
  - API key management handled through existing LLM adapter infrastructure
- Testing strategy:
  - Unit tests: Mock ILLMPort to verify logic and error handling
  - Integration tests: Use test doubles or sandbox LLM endpoints
  - No test should ever make actual paid API calls to external services
- Related documentation:
  - ADR-004: Documents the batching strategy decision and alternatives considered
  - RF-04.6: References the requirement for externalized prompt templates
