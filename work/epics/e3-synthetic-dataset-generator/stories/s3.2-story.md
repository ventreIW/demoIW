# Story: s3.2 — OpenRouter Adapter

## User Story
As a developer,
I want a robust OpenRouter adapter that implements the ILLMPort interface
so that all LLM-dependent features (enrichment, communications, NL query) can share a single, configuration-driven AI gateway without coupling to any specific model.

## Acceptance Criteria

### Scenario: Successful generation
```gherkin
Given an OpenRouterAdapter with valid API key and base URL
When generate is called with a prompt and model
Then it returns the content from the API response
And it logs the model, prompt_tokens, completion_tokens, and latency_ms
```

### Scenario: 4xx error is not retried
```gherkin
Given an OpenRouterAdapter with valid API key and base URL
When generate is called and the API returns a 4xx status
Then it raises ExternalServiceError immediately
And it does not retry
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| Model name provided as argument | Call generate/query | The adapter uses the provided model, not a hardcoded one |
| 503 response on first two attempts, 200 on third | Call generate | The adapter retries twice and returns the successful response |
| Valid API key and base URL in Settings | Initialize adapter | The adapter reads the credentials from Settings |

## Notes
- See s3.2-design.md for technical details.
- The adapter must implement the ILLMPort interface (see app/ports/llm_port.py).
- Environment variables: OPENROUTER_API_KEY, OPENROUTER_BASE_URL.
- Model-specific environment variables (MODEL_DATA_ENRICHMENT, etc.) are used by the caller, not the adapter.
