## Tasks

### T1: Backend tooling baseline (XS) — RISK-FIRST

**Why first:** mypy strict + asyncio pytest config can produce unexpected failures; surface early.

**Files:**
- `backend/pyproject.toml` — Ruff (E,W,F,I,UP, line-length 100), mypy strict + pydantic plugin, pytest (asyncio_mode auto, testpaths = ["tests"])
- `backend/requirements.txt` — fastapi, uvicorn[standard], sqlalchemy[asyncio], asyncpg, alembic, pydantic-settings, httpx, structlog
- `backend/requirements-dev.txt` — pytest, pytest-asyncio, pytest-cov, mypy, ruff, httpx
- `backend/app/__init__.py` — empty (required for mypy strict to find the package)
- `backend/tests/__init__.py` — empty
- `backend/tests/test_placeholder.py` — `def test_placeholder(): assert True`

**TDD cycle:**
- RED: `cd backend && ruff check .` → error (no pyproject.toml config) OR `mypy app/` → error (no package)
- GREEN: Create all files, `pip install -r requirements-dev.txt` → all pass
- REFACTOR: Confirm `asyncio_mode = "auto"` and `pydantic.mypy` plugin load without warnings

**Verification (direct — bootstrapping exception):**
```bash
cd backend && ruff check .
cd backend && mypy app/
cd backend && pytest tests/ -v --tb=short
```

**Done when:** All three commands exit 0. Commit: `feat(s1.1): backend tooling baseline`

## Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| `mypy strict` fails on empty `app/` | Empty `app/__init__.py` gives mypy a valid package; no source = no errors |
| Next.js 15 + `moduleResolution: bundler` breaks typecheck | Use exact tsconfig from design examples; avoid `node16` resolution |
| `pytest-asyncio` version mismatch with `asyncio_mode = "auto"` | Pin `pytest-asyncio>=0.23` in requirements-dev.txt |
| `npm test -- --run` exits non-zero if no test files found | Include `frontend/src/placeholder.test.ts` before running test command |
| CI yaml syntax error blocks PR merges | Validate with `actionlint` locally before pushing |
