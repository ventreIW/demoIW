## What went well

- **Design doc was accurate** — the file structure, imports, and patterns in `s1.3-design.md` mapped cleanly to implementation. Zero structural surprises.
- **TDD cycle worked end-to-end** — wrote the failing test first, implemented the route, test passed. The DB-unavailable test exposed a dependency-injection edge case immediately (see improvements).
- **SQLite in-memory for CI** — using `sqlite+aiosqlite://` for tests means no PostgreSQL dependency in CI, yet the code paths through real SQLAlchemy execute paths are exercised identically.
- **ruff/mypy gate was fast** — both passed on first run after ruff auto-fix, with only a single manual fix needed (lifespan return type annotation).

## What to improve

- **`client_unavailable` fixture design** — the design doc showed raising `RuntimeError` inside the dependency override, but FastAPI catches exceptions during dependency resolution and returns 500, bypassing the health endpoint's try/except entirely. The fix (bad DB URL that fails at `.execute()` time, not at session creation) should be documented as a pattern.
- **`orjson` missing from requirements** — `main.py` used `ORJSONResponse` but `orjson` wasn't in `requirements.txt`. Tests failed on first run. Added at install time; needs to be in the dependency file long-term.
- **`app/core/` vs design spec mismatch** — pre-existing code had `app/core/config.py` and `app/core/db.py` but the design spec said `app/config.py` and `app/infrastructure/`. This was caught during the restructuring pass. Would have been avoided if the skeleton had been built fresh from the design spec rather than incrementally.

## Heutagogical Checkpoint

1. **What did I learn?**
   - FastAPI dependency overrides that raise during resolution produce 500 errors (unhandled server errors), not exceptions that reach the route handler's try/except. To test database-unavailable scenarios, the session must be created successfully but fail at query time — inject a bad connection URL, not a raising callable.
   - `AsyncGenerator[None, None]` is the correct return type annotation for FastAPI lifespan functions when using `@asynccontextmanager`.
   - `aiosqlite` is needed for `sqlite+aiosqlite://` URLs with async SQLAlchemy — it's a separate pip package, not bundled with SQLAlchemy's asyncio extra.

2. **What would I change about the process?**
   - Add `orjson` to `requirements.txt` explicitly — it's a transitive dependency of FastAPI for `ORJSONResponse` but not guaranteed to be installed.
   - When the design specifies a file layout, implement it directly rather than adapting existing code that uses a different layout. The restructuring pass was avoidable overhead.

3. **Framework improvements?**
   - The `rai-story-design` template should include a "test fixture strategy" section — the `client_unavailable` edge case (DI vs. handler scope) is subtle and recurring.

4. **More capable of now?**
   - FastAPI app factory + lifespan pattern from scratch
   - Async SQLAlchemy 2.0 with dependency injection
   - Pytest async fixtures with httpx ASGI transport
   - Alembic configured for async engines
   - Structlog conditional renderer (dev console vs. production JSON)
