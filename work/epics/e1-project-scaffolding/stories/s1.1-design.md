# s1.1 Design — Repository Scaffold & CI Pipeline

**Complexity:** Simple  
**Branch:** `story/s1.1/repo-scaffold`

---

## Problem

The monorepo has no `frontend/` or `backend/` directories and no CI pipeline. Every developer works against different implicit tooling assumptions, and there is no automated gate to catch quality regressions before merge.

## Value

Establishes a shared, enforced quality baseline: strict TypeScript, consistent formatting, and a CI pipeline that blocks bad merges. Every subsequent story builds on this foundation — a broken one costs the whole team time.

---

## Approach

Create the directory skeleton and tooling config files for both apps plus a GitHub Actions CI workflow. All output is static config — no application logic, no external services, no DB.

**Lean gates:**
- **KISS**: Config-only. No abstractions, no scripts beyond what CI needs.
- **DRY**: Each config file is in its canonical location; no duplication across apps.
- **YAGNI**: No Docker, no deployment config, no secrets management (`.env.example` already exists; `.env.example` with real fields comes in s1.3).
- **MVP**: Minimum set of files that makes both apps pass their own quality checks in CI.

**Supply chain:** All tooling packages (Next.js, TypeScript, Vitest, ESLint, Prettier, Ruff, mypy, pytest, FastAPI, SQLAlchemy, Alembic, Pydantic) are well-known, widely adopted — supply chain health is established. No new niche dependencies.

**Legacy sweep:** All files in `frontend/`, `backend/`, and `.github/` are net-new. Root `.gitignore` is expanded (not replaced) with Node/Python artifact patterns. Nothing becomes orphaned.

**Drift risks:** None — fanout is contained to 3 leaf directories with no coupling edges.

## Components

| File | Change | Purpose |
|------|--------|---------|
| `.gitignore` | modify | Add Node/Python artifact patterns to existing RaiSE entries |
| `frontend/package.json` | create | Project metadata + scripts: `dev`, `build`, `typecheck`, `lint`, `test` |
| `frontend/tsconfig.json` | create | TypeScript strict mode, `@/*` path alias, `moduleResolution: bundler` |
| `frontend/.eslintrc.json` | create | Next.js recommended + TypeScript rules |
| `frontend/prettier.config.js` | create | Single quotes, no semis, Tailwind plugin |
| `frontend/vitest.config.ts` | create | jsdom environment, coverage thresholds |
| `backend/pyproject.toml` | create | Ruff (E,W,F,I,UP, line-length 100), mypy strict, pytest (asyncio_mode auto) |
| `backend/requirements.txt` | create | Production deps: fastapi, uvicorn, sqlalchemy, asyncpg, alembic, pydantic-settings, httpx, structlog |
| `backend/requirements-dev.txt` | create | Dev deps: pytest, pytest-asyncio, pytest-cov, mypy, ruff, httpx |
| `.github/workflows/ci.yml` | create | Parallel frontend + backend jobs on PR to `main` |
| `README.md` | modify | Add local dev setup instructions for both apps |

## Examples

### CI workflow

```yaml
# .github/workflows/ci.yml
name: CI
on:
  pull_request:
    branches: ["main"]

jobs:
  frontend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: frontend
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20", cache: "npm", cache-dependency-path: frontend/package-lock.json }
      - run: npm ci
      - run: npm run typecheck
      - run: npm run lint
      - run: npm test -- --run

  backend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: backend
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements-dev.txt
      - run: ruff check .
      - run: mypy app/
      - run: pytest tests/ -v --tb=short
```

Note: backend CI at this stage has no real DB — the first pytest target is a placeholder `tests/test_placeholder.py` that just asserts `True`. The DB connection test comes in s1.3.

### `frontend/tsconfig.json` (critical fields)

```json
{
  "compilerOptions": {
    "strict": true,
    "paths": { "@/*": ["./src/*"] },
    "moduleResolution": "bundler",
    "target": "ES2022",
    "lib": ["dom", "dom.iterable", "esnext"],
    "jsx": "preserve",
    "incremental": true,
    "plugins": [{ "name": "next" }]
  },
  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx", ".next/types/**/*.ts"],
  "exclude": ["node_modules"]
}
```

### `backend/pyproject.toml` (critical sections)

```toml
[tool.ruff]
line-length = 100
select = ["E", "W", "F", "I", "UP"]

[tool.mypy]
strict = true
plugins = ["pydantic.mypy"]

[tool.pytest.ini_options]
asyncio_mode = "auto"
testpaths = ["tests"]
```

## Acceptance Criteria

**MUST:**
- `cd frontend && npm ci && npm run typecheck` exits 0 with zero TypeScript errors
- `cd frontend && npm run lint` exits 0 with zero ESLint violations
- `cd frontend && npm test -- --run` finds and passes at least one test (placeholder)
- `cd backend && pip install -r requirements-dev.txt && ruff check . && mypy app/` all exit 0
- `cd backend && pytest tests/ -v` passes at least one test (placeholder `assert True`)
- `.github/workflows/ci.yml` YAML is syntactically valid (verified by `yamllint` or `actionlint`)
- `README.md` contains working local dev setup steps for both apps

**SHOULD:**
- `frontend/package.json` uses exact versions (no `^`) for reproducible installs
- CI jobs are parallel (not sequential) — both `frontend` and `backend` appear as sibling jobs

**MUST NOT:**
- No application source files (`app/`, `src/app/`) — config only
- No Docker or docker-compose files
- No real `.env` committed (`.env.example` already exists)

## Testing strategy

| Layer | Name | Purpose |
|-------|------|---------|
| CI syntax | `actionlint` on `ci.yml` | Catch YAML schema errors before push |
| Frontend | `npm run typecheck` | Zero-error baseline on empty src |
| Frontend | `npm run lint` | Config is correct and rule set loads |
| Backend | `ruff check . && mypy app/` | Config loads without crashes |
| Backend | `pytest tests/test_placeholder.py` | pytest finds tests and runner works |
