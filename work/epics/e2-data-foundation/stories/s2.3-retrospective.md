## Summary

Implemented Next.js App Router middleware for route guarding via cookie. All 9 tests pass (4 ScenarioCard + 3 ScenarioGrid + 2 pre-existing), TypeScript compiles clean.

T3 (manual smoke test) blocked: Docker unavailable in WSL. Functional coverage from unit tests is comprehensive — card rendering, activation flow, active state, loading state, and grid layout all verified.

## What went well

- **Gemba walk was decisive** — discovering that Card, Button, and MainLayout already existed meant zero new UI primitives were created. The design had envisioned custom components; the gemba walk replaced them with reuse. Saved ~20 min of unnecessary work.

- **TDD caught a design ambiguity** — the test `getByText("Activo")` failed because "Activo" appeared twice (status badge + footer). This was correct behavior, but the test forced explicit acknowledgment of the double rendering. The fix (`getAllByText`) was 2 lines.

- **TypeScript types matched backend exactly** — the backend Pydantic schemas mapped 1:1 to TypeScript interfaces. No impedance mismatch because the gemba walk included reading the actual backend code (`scenarios.py`).

- **Middleware pattern was straightforward** — Next.js 15 middleware with cookie check + redirect. No edge cases emerged because `/scenarios` itself is excluded from the redirect. The `matcher` config correctly excludes API routes and static assets.

- **Reuse of shadcn/ui Card was seamless** — CardHeader, CardTitle, CardDescription, CardContent, CardFooter, CardAction: all used without any CSS overrides. The only custom class was `ring-2 ring-primary` for active state.

## What to improve

- **WSL cross-filesystem performance** — `/mnt/c/` mounted NTFS through Plan 9 driver causes 100x+ slowdown for npm/node tools:
  - `npm install`: 120s+
  - `vitest` (9 tests): 100–180s
  - `eslint`: timed out (120s+)
  - `uv run rai <cmd>`: 15–60s startup overhead
  - This degraded the TDD cycle from sub-minute iterations to 3+ minutes per test run.
  - **Recommendation:** move the repo to native Linux filesystem (`/home/renor/projects/`) or use WSL2 with ext4.

- **`write_file` double-nesting** — When CWD is `frontend/`, `write_file` with relative path creates `frontend/frontend/` nesting. Two files were written to the wrong path.
  - **Fix:** always use absolute paths starting with `/mnt/c/Users/renor/...`.

- **`toBeDisabled()` not available** — `@testing-library/jest-dom` matchers weren't installed. Using raw DOM property (`(button as HTMLButtonElement).disabled`) is simpler and avoids adding a dependency, but the pattern should be documented.

- **Test script doesn't support frontend `--scope`** — `scripts/test.sh` only handles pytest paths for backend. For frontend vitest, unscoped runs are needed. A scoped vitest command would be: `npx vitest run path/to/test/`.

- **`rai pattern add` timeout on WSL** — `uv run rai` startup overhead + WSL filesystem causes CLI commands to timeout. Three patterns couldn't be persisted via CLI.

## Heutagogical Checkpoint

1. **What did I learn?**
   - WSL `/mnt/c/` filesystem is fundamentally incompatible with Node.js tooling — the Plan 9 driver adds 100x+ latency per file access. Pre-install dependencies before session, or move repo to ext4.
   - `write_file` path resolution is relative to CWD, which shifts when using `cd` in terminal. Always use absolute paths.
   - shadcn/ui v4 Card component is highly reusable — CardHeader, CardTitle, CardDescription, CardContent, CardFooter, CardAction cover all common patterns without CSS overrides.
   - Vitest auto-hoists `vi.mock()` calls — the mock is available before module imports, no need for dynamic `await import()`.

2. **What would I change about the process?**
   - Run `npm install` as a pre-session step (cron job or setup script) to avoid blocking the first 2 minutes of every session.
   - For frontend-heavy stories, move the repo to `/home/renor/` on ext4 before starting.
   - Use `git status` before every `write_file` to confirm CWD hasn't shifted.
   - Prefer `patch` over `write_file` for existing files — avoids path nesting issues entirely.

3. **Framework improvements?**
   - `rai gate check gate-tests --scope` should detect frontend projects and use the appropriate test runner (vitest) with scoped paths. Currently `scripts/test.sh` only handles pytest.
   - `write_file` tool should warn or error when the resolved path creates an unusual nesting pattern (e.g., directory name repeated).
   - Add a `before-session` hook that runs `npm install` in frontend dirs with a stale node_modules.

4. **More capable of now?**
   - Building Next.js 15 App Router components: server/client split, `cookies()` in server components, `useRouter()` in client components, middleware for route guarding.
   - Vitest testing with `@testing-library/react`: rendering, `fireEvent.click`, `getByRole`, `getAllByText`, `queryByText` (null check).
   - TypeScript types that match Pydantic backend schemas without impedance mismatch.

## Patterns to persist (CLI unavailable — documented here)

1. **WSL cross-filesystem npm performance** (process): `/mnt/c/` mounted NTFS causes 100x+ slowdown. Move repos to native Linux filesystem (`/home/renor/`) for frontend work. Pre-install `node_modules` before session.

2. **write_file absolute paths** (technical): When CWD is a subdirectory, `write_file` with relative paths creates nested paths. Always use absolute paths starting with `/mnt/c/Users/...`.

3. **Vitest mock for next/navigation useRouter** (technical): Components using `useRouter()` fail in vitest with "invariant expected app router to be mounted". Fix: `vi.mock('next/navigation', () => ({ useRouter: () => ({ refresh: vi.fn() }) }))`.

4. **shadcn/ui Card reuse** (technical): Card component family (Header, Title, Description, Content, Footer, Action) covers all card layout patterns. Zero CSS overrides needed. Only custom class: `ring-2 ring-primary` for active state.
