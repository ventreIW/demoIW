## What to improve

1. **Design code examples used wrong `with_variant` syntax** — the design doc showed `Text().with_variant("jsonb", "postgresql")` as the approach, but didn't flag that both args must be SQLAlchemy types, not strings. Should have shown `JSON` directly in the design example (which is what was actually needed). Fix: design examples must be runnable, not illustrative.
2. **Timezone handling should have been specified in design, not discovered at GREEN** — the design noted "SQLite type incompatibility" as a risk but didn't specify the mitigation. The `_utc()` pattern was obvious once hit, but would have been better as an explicit design decision.
3. **Multiple ruff fix passes** — needed 3 separate `ruff check --fix` runs (domain, adapters, migration files). A habit of running `ruff check . --fix` immediately after each file creation would eliminate this overhead.
4. **`gate check` from wrong directory twice** — called `rai gate check` from `backend/` instead of repo root, getting "No .raise/manifest.yaml found". Root is the correct CWD for gate commands.

## Heutagogical Checkpoint

1. **What did I learn?**
   - `SQLAlchemy.Type.with_variant(other, dialect)` — both positional args are SQLAlchemy type objects. `"jsonb"` as the first arg is a string, not a type, causing `AttributeError: 'str' object has no attribute '_variant_mapping'` at ORM import time.
   - SQLite stores datetimes as naive strings (no tzinfo). When reading back, the column is a naive `datetime`. PostgreSQL stores `TIMESTAMP WITH TIME ZONE` and returns timezone-aware datetimes. The mapper must normalize this: `_utc(dt)` attaches UTC when tzinfo is None.
   - `rai gate check` must be run from the repo root (where `.raise/manifest.yaml` lives), not from a subdirectory.
   - Alembic `env.py` imports `Base` by module path; the module must be importable and `Base` must exist before `alembic upgrade head` can run — even if no ORM models are defined yet.

2. **What would I change about the process?**
   - Design examples for ORM columns must use the exact import form that will compile, not approximate pseudocode. One test of "does this import?" at design time would have caught `with_variant`.
   - Add timezone handling explicitly to the design's "test fixture strategy" section whenever `datetime` fields and SQLite are both involved.
   - Run `ruff check . --fix` as part of every file creation step, not as a cleanup pass at the end.

3. **Framework improvements?**
   - The `rai-story-design` skill should add: "If the story uses SQLite for tests and PostgreSQL in production, document the dialect-specific types (UUID, JSONB, TIMESTAMP WITH TIME ZONE) and their SQLite equivalents in the design's components table."
   - The plan gate policy section should note: "`rai gate check` must be run from the repo root (where `.raise/manifest.yaml` is located)."

4. **More capable of now?**
   - Full SQLAlchemy 2.0 mapped-column ORM layer with cross-dialect compatibility
   - Alembic migration authoring with `op.create_table`, FK constraints, partial unique indexes, and data migrations
   - Pydantic v2 frozen models with StrEnum fields and `ValidationError` behavior
   - Bidirectional ORM ↔ domain mapper pattern with `_utc()` normalization for SQLite CI
