## Design doc adjustments

Antes de implementar, el design doc tiene 3 discrepancias con el código real de s2.1:

| Issue | Design dice | Realidad | Acción |
|-------|-------------|----------|--------|
| URL prefix | `/api/scenarios` | El proyecto usa `/api/v1` (`API_V1_STR`) | Usar `/api/v1/scenarios` |
| `client_count` | Campo en `ScenarioSummary` | No es columna en `ScenarioORM` — requiere `COUNT(*)` sobre `clients` | Calcular con subquery en el repo |
| POST response | `response_model=ScenarioDetail` | La story AC dice "returns its summary" | POST devuelve `ScenarioSummary`, no `ScenarioDetail` |

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|:----------:|:------:|------------|
| `client_count` requiere JOIN/COUNT — performance implications | Low | S1 uses SQLite in-memory, no perf issue | OK for current seed data (3 scenarios); revisit if >100 scenarios |
| Route `/active` vs `/{scenario_id}` ordering bug | Medium | Block T6 | Declare `/active` before `/{scenario_id}` — design doc ya lo advierte |
| `API_V1_STR` = `/api/v1` no está en el design doc | High | Todos los tests fallan por URL incorrecta | Documentado en los ajustes del plan; usar `/api/v1/scenarios` |
| `container.py` no existe — hay que crearlo desde cero | Low | T1 | La health check inyecta `AsyncSession` directo; el container es nuevo pero simple |
| UUID como string en SQLite vs UUID nativo en PostgreSQL | Medium | Consultas por ID pueden fallar | ScenarioORM.id es `String(36)` — consistente con los mappers existentes |
