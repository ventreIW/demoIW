## Summary

Built the Scenario Management API with 5 endpoints (list, create, detail, activate, get-active), a repository pattern with DI, and 9 integration tests. All ACs met. The API is ready for s2.3 (frontend) and s2.4 (CSV upload).

---

## What went well

- **TDD cycle was clean** — RED→GREEN→REFACTOR en cada tarea. Los tests atraparon 3 bugs antes del commit: sector inválido en el test (`fintech` no existe), falta de `commit()` en el repo, y ruta `/active` capturada por `/{scenario_id}`.
- **Repository pattern encajó naturalmente** — `IScenarioRepository` (ABC) + `SQLAlchemyScenarioRepository` + `container.py` (DI) siguen la misma estructura que el health endpoint ya usaba (`Depends(get_session)`). Sin fricción arquitectónica.
- **Design doc review pre-implementación detectó 3 discrepancias** — URL prefix (`/api` vs `/api/v1`), `client_count` como campo calculado, y POST response type. Corregidas antes de codear, zero rework.
- **Route ordering fix fue inmediato** — el test RED para `/active` devolvió 422 en vez de 404, confirmando exactamente el problema que el design doc advertía. La solución (mover `/active` antes de `/{scenario_id}`) tomó 2 minutos.

---

## What to improve

- **`flush()` vs `commit()` en SQLite in-memory** — El repo usaba `flush()` asumiendo que los datos serían visibles en la misma transacción. Pero SQLite in-memory (`sqlite+aiosqlite://`) crea una DB nueva por conexión, y cada request de test obtiene una sesión nueva. `flush()` no persiste entre sesiones. Se cambió a `commit()` en `add()` y `set_active()`. Esto debería documentarse como PAT.
- **`client` fixture no creaba tablas** — El fixture existente solo creaba el engine; `Base.metadata.create_all` estaba solo en `db_session`. Los tests de integración que usan `client` necesitan tablas. Se agregó `create_all` al fixture. Esto no rompió los tests existentes (el health check usa `SELECT 1`, los mapper tests usan `db_session`).
- **Plan estimó 125 min, real ~110 min** — T1 y T5 tomaron menos de lo esperado porque el patrón de DI y el repo ya estaban bien diseñados en el design doc.

## Heutagogical Checkpoint

1. **What did I learn?**
   - SQLite in-memory con `aiosqlite` crea una DB independiente por conexión. `session.flush()` no es suficiente para tests de integración multi-request; se requiere `session.commit()`.
   - FastAPI resuelve rutas en orden de declaración. Rutas literales (`/active`) deben ir antes que las parametrizadas (`/{id}`) para evitar captura.
   - El patrón `IScenarioRepository` (ABC) + implementación concreta + `Depends` para DI escala bien. Misma estructura funcionará para `IClientRepository`, `IInvoiceRepository`, etc.
   - La revisión del design doc contra el código real de s2.1 (modelos ORM, mappers) antes de implementar evitó 3 discrepancias que habrían causado rework.

2. **What would I change about the process?**
   - Agregar un paso de "verify test fixtures create necessary DB state" al plan. El `client` fixture sin `create_all` fue una sorpresa que costó ~10 min diagnosticar.
   - Documentar el comportamiento de SQLite in-memory con `aiosqlite` como PAT para futuros tests de integración.

3. **Framework improvements?**
   - El `rai gate check` falla en gates de frontend (`eslint`, `vitest`, `tsc`) porque `npm install` no se ejecutó. Esto es un problema de environment, no de código. Considerar un `--scope backend` para gate check.

4. **More capable of now?**
   - FastAPI + SQLAlchemy async repository pattern with DI.
   - Integration testing with SQLite in-memory across multiple HTTP requests.
   - Route ordering and FastAPI path parameter capture semantics.
   - Detecting and fixing design doc discrepancies before implementation.
