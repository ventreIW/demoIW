**conftest.py addition (`db_session` fixture):**

from app.infrastructure.database import Base

@pytest.fixture
async def db_session() -> AsyncGenerator[AsyncSession, None]:
    engine = create_async_engine("sqlite+aiosqlite://", echo=False)
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| PostgreSQL not running locally for T3 | Medium | `pg_isready` check before T3; migration can be written/reviewed without live DB, only verify step needs it |
| SQLite type incompatibility with `PG_UUID` or `JSONB` in T2 mapper tests | Medium | Use `String` for UUID and `JSON` (not JSONB) in ORM models when dialect is SQLite, or use `TypeDecorator` — simpler: test with SQLite, migration uses PostgreSQL-native types |
| `mypy` strict on 14 mapper functions | Low | Use explicit type annotations on all function signatures; mappers are pure data conversion, no branching |
| `down_revision` chain error in migration | Low | Verify chain: 0001 → 0002 → 0003; Alembic will refuse to run if chain is broken |

> Note on SQLite compatibility for T2: `JSONB` and `PG_UUID` are PostgreSQL-specific. In ORM models, use `sa.String` for UUIDs and `sa.JSON` for the parameters column — both are SQLite-compatible. The migration (T3) uses `postgresql.JSONB` explicitly via `op.create_table`. The ORM column type and the migration type can differ: ORM uses cross-dialect `JSON`, migration targets PostgreSQL only.
