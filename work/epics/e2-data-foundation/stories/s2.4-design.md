**Complexity:** Simple  
**Gemba walk:** Complete ‚Äî 2026-06-29  
**Dependencies:** s2.2 (backend API) ‚úì complete, s2.3 (scenario selector) ‚úì complete

## Problem

The scenario selector page has a disabled "Subir CSV" button stub. Users cannot upload CSV files to create scenarios. The backend endpoint `POST /api/v1/scenarios/upload-csv` is being built by Nano in parallel ‚Äî this story builds the frontend against the agreed contract.

## Value

A demo facilitator can create scenarios by uploading CSV files directly from the scenario selector. This enables batch data import without manual API calls. After upload, the new scenario appears in the grid and is immediately selectable.

---

## Approach

Frontend-only story. The backend endpoint `POST /api/v1/scenarios/upload-csv` is Nano's responsibility ‚Äî built in parallel against the same contract. This story delivers the frontend side: API client, file picker component, and page integration.

A **client component** (`CsvUpload`) replaces the disabled "Subir CSV" stub button with a functional file picker + upload flow. On file selection, it calls `uploadCsv()` (new function in the API client), sends the file as `multipart/form-data`, and displays success/error from the backend response. On success, `router.refresh()` reloads the scenario grid (same pattern as activation flow).

- **`lib/api/scenarios.ts`** ‚Äî existing `API_BASE` constant, typed fetch pattern (`listScenarios`, `activateScenario`). Add `uploadCsv()` following same shape.
- **`ScenarioSummary` type** ‚Äî already defined in `frontend/src/types/scenario.ts`. Return type for `uploadCsv()`.

- **Button styles** ‚Äî `page.tsx:39-50` uses `inline-flex h-8 items-center justify-center rounded-lg` pattern. Match for "Subir CSV" button.

### Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Client component for file picker | Next.js server components cannot handle `<input type="file">` ‚Äî must be client-side. Same pattern as `ScenarioGrid`. |
| D2 | `router.refresh()` after upload | Same pattern as activation flow (D2 in s2.3 design). Simpler than client-state re-fetch. |
| D3 | Client wrapper (`CsvUploadWrapper`) for router injection | `page.tsx` is a server component ‚Äî can't use `useRouter`. Small wrapper keeps separation clean. |
| D4 | Mock `uploadCsv` in tests, no backend needed | Backend is Nano's responsibility ‚Äî built in parallel. Vitest mocks cover all states. |
| D5 | No CSV validation on frontend | Backend validates and returns 422 with field-level errors. Frontend is a dumb pipe ‚Äî displays whatever backend returns. |

## Examples

### 1. Backend contract (assumed ‚Äî Nano's endpoint)

```typescript
// src/lib/api/scenarios.ts ‚Äî new function
export async function uploadCsv(file: File): Promise<ScenarioSummary> {
  const formData = new FormData()
  formData.append("file", file)
  const res = await fetch(`${API_BASE}/api/v1/scenarios/upload-csv`, {
    method: "POST",
    body: formData,
  })
  if (!res.ok) {
    const err = await res.json()
    const messages = err.detail?.map((d: {msg: string}) => d.msg).join(", ")
    throw new Error(messages || `Upload failed: ${res.status}`)
  }
  return res.json()
}
```

## Acceptance Criteria

### MUST
- [ ] "Subir CSV" button opens a native file picker filtered to `.csv` files
- [ ] Selecting a file shows the file name and enables an upload action
- [ ] Upload sends file as `multipart/form-data` to `POST /api/v1/scenarios/upload-csv`
- [ ] Successful upload shows a success message and triggers scenario grid refresh
- [ ] Backend validation errors (422) are displayed as error messages in the UI
- [ ] No TypeScript errors (`cd frontend && npm run typecheck`), no lint errors (`cd frontend && npm run lint`)
- [ ] All frontend tests pass (`cd frontend && npm test -- --run`)

### SHOULD
- [ ] Upload button shows loading state while request is in flight
- [ ] Network errors (backend unreachable) show a user-friendly error message

### MUST NOT
- [ ] No new external dependencies added to `frontend/package.json`
- [ ] No full page reload on upload (use `router.refresh()`)
- [ ] No CSV validation on frontend (backend is the single source of truth for validation)
